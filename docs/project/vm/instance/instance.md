# 인스턴스

CloudiA의 `인스턴스` 메뉴에서는 가상머신을 생성하고, 상태를 확인하고, 전원과 스냅샷, 연결된 리소스를 관리합니다. 이 문서는 실제 UI 사용 절차를 먼저 설명하고, 운영자 성격의 고급 내용은 뒤쪽 `운영 참고`와 `부록`으로 분리합니다.

## 인스턴스 개요

### 메뉴에서 할 수 있는 일

- 프로젝트의 인스턴스 목록을 확인합니다.
- 인스턴스를 생성하고 편집합니다.
- 인스턴스를 실행, 종료, 재부팅, 삭제합니다.
- 상세 화면에서 네트워크, 디스크, 스냅샷, 연결 리소스, 모니터링 정보를 확인합니다.
- 인스턴스 스냅샷을 생성, 편집, 삭제, 롤백합니다.

### 화면 구성

| 화면 | 설명 |
| --- | --- |
| `목록` | 인스턴스 상태를 확인하고 생성, 편집, 전원 제어, 삭제, 스냅샷 생성을 시작합니다. |
| `생성` | 이름, 스펙, 설치 유형, 네트워크, 사용자 계정, 선택 설정을 입력해 인스턴스를 만듭니다. |
| `편집` | 생성과 같은 화면 구성을 사용합니다. 다만 생성 후 변경할 수 없는 항목은 잠기며, 실행 중에는 일부 항목만 수정할 수 있습니다. |
| `상세` | 현재 상태와 연결된 리소스를 확인하고, 디스크 부팅 순서 변경, 스냅샷 관리 같은 후속 작업을 수행합니다. |

### 상태 이해하기

| 상태 | 의미 | 사용자 안내 |
| --- | --- | --- |
| `생성 중` | 인스턴스 생성 작업이 진행 중입니다. | 작업이 끝날 때까지 기다립니다. |
| `부팅 중` | 전원이 인가되고 게스트 OS가 올라오는 중입니다. | 상태가 안정될 때까지 기다립니다. |
| `실행 중` | 게스트 OS가 동작 중입니다. | 콘솔 접속과 전원 제어가 가능합니다. |
| `재부팅 중` | 재부팅 작업이 진행 중입니다. | 작업이 끝날 때까지 기다립니다. |
| `종료 중` | 종료 작업이 진행 중입니다. | 작업이 끝날 때까지 기다립니다. |
| `종료됨` | 전원이 꺼진 상태입니다. | `실행`할 수 있습니다. |
| `정지됨` | 전원이 꺼진 상태입니다. | `실행`할 수 있습니다. |
| `일시 중지` | 실행 상태가 일시 중지되었습니다. | 일부 작업이 제한됩니다. |
| `마이그레이션 중` | 호스트 간 이동 작업이 진행 중입니다. | 작업이 끝날 때까지 기다립니다. |
| `삭제 중` | 삭제 작업이 진행 중입니다. | 완료될 때까지 기다립니다. |
| `에러` | 생성 또는 운영 중 오류가 발생했습니다. | 오류 원인을 먼저 확인합니다. |

> 중요
>
> `종료됨`과 `정지됨`은 모두 전원이 꺼진 상태이지만, 내부 상태 정의는 다릅니다.
>
> - `종료됨`: 현재 어떤 호스트에도 배치되어 있지 않은 종료 상태
> - `정지됨`: 시스템 내부에서 정지 상태로 관리되는 종료 상태
>
> 사용자 관점에서는 두 상태 모두 `실행`할 수 있습니다. 운영 판단이나 장애 분석이 필요할 때는 상태명을 구분해 확인합니다.

> 참고
>
> 이름을 클릭해 상세 화면에 들어갈 수 없는 상태는 `생성 중`, `삭제 중`, `에러`입니다.

### 관리 대상 구분

| 생성 방식 | 인스턴스 메뉴에서 할 일 |
| --- | --- |
| `수동 생성` | 이 메뉴에서 생성, 편집, 전원 제어, 삭제, 스냅샷 작업을 수행합니다. |
| `오토 스케일링 그룹`, `쿠버네티스`, `파일 시스템`, `Gen:iA` 등 | 이 메뉴에서는 주로 상태와 상세 정보만 확인합니다. 실제 관리는 상위 서비스 화면에서 수행합니다. |

> 중요
>
> 상위 서비스에서 생성된 인스턴스는 인스턴스 메뉴에 작업 버튼이 표시되더라도 비활성화될 수 있습니다. 이런 인스턴스는 원래 생성한 서비스 화면에서 관리합니다.

## 목록에서 할 수 있는 작업

### 목록에서 먼저 확인할 항목

| 항목 | 확인할 내용 |
| --- | --- |
| `이름` | 인스턴스 표시 이름입니다. 이름을 클릭하면 상세 화면으로 이동합니다. |
| `생성 방식` | `수동 생성`인지, 상위 서비스에서 생성된 인스턴스인지 확인합니다. |
| `IP 주소` | NIC에 할당된 IPv4 주소를 표시합니다. NIC가 여러 개면 쉼표로 구분해 함께 표시합니다. |
| `상태` | 현재 전원 상태와 작업 가능 범위를 확인합니다. |
| `vCPU`, `메모리` | 현재 할당된 기본 스펙을 확인합니다. |
| `생성 일시` | 생성 시점을 확인합니다. |

> 참고
>
> 목록의 `IP 주소` 컬럼에서 가장 앞에 표시되는 IPv4 주소는 기본 네트워크 인터페이스의 IPv4 주소입니다.

### 검색과 정렬

- 검색 조건은 `이름`입니다.
- `IP 주소` 컬럼은 정렬되지 않습니다.
- 기본 정렬은 `생성 일시` 기준입니다.
- 페이지당 10개를 표시합니다.
- 목록에서는 좌측 라디오 버튼으로 인스턴스 1개만 선택할 수 있습니다.

### 작업 버튼 사용 기준

아래 표는 목록 화면의 대표 버튼을 같은 기준으로 정리한 것입니다.

- `버튼 표시`: 버튼이 화면에 보이는지
- `버튼 활성화`: 현재 선택한 인스턴스 기준으로 버튼을 클릭할 수 있는지
- `실제 작업 성공`: 버튼 클릭 후 작업이 실제로 완료되기 위한 조건

| 작업 | 버튼 표시 | 버튼 활성화 조건 | 실제 작업 성공 조건 |
| --- | --- | --- | --- |
| `생성` | 항상 표시 | 항상 활성화 | 필수 입력값과 자원 검증을 통과해야 합니다. |
| `편집` | 항상 표시 | 인스턴스 1개를 선택하고, `수동 생성`이며, 상태가 `종료됨`, `정지됨`, `실행 중`이어야 합니다. 단, `ISO 파일` 설치 유형은 `실행 중`일 때 비활성화됩니다. | 편집 화면에서 수정 가능한 항목만 저장할 수 있습니다. 실행 중 편집은 라이브 업데이트 가능 항목만 반영됩니다. |
| `실행` | 항상 표시 | 인스턴스 1개를 선택하고, `수동 생성`이며, 상태가 `종료됨` 또는 `정지됨`이어야 합니다. | 호스트 배치 조건과 연결 리소스 조건을 만족해야 합니다. |
| `종료` | 항상 표시 | 인스턴스 1개를 선택하고, `수동 생성`이며, 상태가 `실행 중` 또는 `일시 중지`여야 합니다. | 종료 요청이 정상 처리되어야 합니다. |
| `재부팅` | 항상 표시 | 인스턴스 1개를 선택하고, `수동 생성`이며, 상태가 `실행 중`이어야 합니다. | 보안 유형과 시스템 정책에서 재부팅을 허용해야 하며, 재부팅 요청이 정상 처리되어야 합니다. |
| `삭제` | 항상 표시 | 인스턴스 1개를 선택하고, `수동 생성`이어야 합니다. `생성 중`, `실행 중`, `부팅 중`, `재부팅 중`, `종료 중`, `삭제 중` 상태에서는 비활성화됩니다. `에러` 상태는 원인 확인이 끝날 때까지 제한될 수 있습니다. | 연결된 인스턴스 스냅샷이 0개여야 합니다. |
| `더보기 > 스냅샷 생성` | 메뉴에 표시 | 인스턴스 1개를 선택하고, 상태가 `실행 중` 또는 `종료됨`이어야 합니다. 진행 중인 작업이 있으면 비활성화될 수 있습니다. | 스냅샷 이름 검증과 내부 처리 조건을 통과해야 합니다. |

> 참고
>
> 이 문서에서 `조건부`는 현재 상태는 허용되지만 추가 조건을 더 확인해야 함을 뜻합니다. `제한`은 상태 외에 생성 방식, 오류 상태, 상위 서비스 정책 때문에 인스턴스 메뉴에서 바로 처리할 수 없음을 뜻합니다.

| 상태 | 편집 | 실행 | 종료 | 재부팅 | 삭제 | 스냅샷 생성 | 안내 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| `실행 중` | 가능<sup>1</sup> | 불가 | 가능 | 가능 | 불가 | 가능 |  |
| `종료됨` | 가능 | 가능 | 불가 | 불가 | 조건부 | 가능 |  |
| `정지됨` | 가능 | 가능 | 불가 | 불가 | 조건부 | 불가 |  |
| `일시 중지` | 제한 | 불가 | 가능 | 불가 | 조건부 | 불가 | 상태 해제 또는 종료 후 다시 시도합니다. |
| `생성 중`, `부팅 중`, `재부팅 중`, `종료 중`, `삭제 중` | 불가 | 불가 | 불가 | 불가 | 불가 | 불가 | 진행 중인 작업이 끝난 뒤 다시 시도합니다. |
| `에러` | 제한 | 제한 | 제한 | 제한 | 제한 | 제한 | 오류 원인을 먼저 확인합니다. |

<sup>1</sup> `ISO 파일` 설치 유형의 인스턴스는 `실행 중`일 때 편집할 수 없습니다.

## 인스턴스 생성

### 생성 전에 준비할 항목

- 인스턴스가 연결될 `VPC`를 준비합니다.
- 각 NIC가 연결될 `서브넷`을 준비합니다.
- 접속 방식을 `비밀번호`, `SSH 키`, 또는 둘 다 사용할지 결정합니다.
- `OS 이미지`로 만들지, `ISO 파일`로 직접 설치할지 결정합니다.
- 추가 `블록`, `파일 시스템`, `AI 가속기`, `사용자 스크립트`가 필요한지 확인합니다.

### 1. 기본 정보 입력

- `인스턴스 이름`을 입력합니다.
- 스펙은 `인스턴스 타입 선택` 또는 `직접 입력`으로 지정합니다.
- 필요하면 `vCPU 상세 설정`에서 하이퍼스레딩 사용 여부와 코어 수를 조정합니다.

| 항목 | 입력 기준 | 비고 |
| --- | --- | --- |
| `인스턴스 이름` | 영문 대/소문자, 숫자, `-`, `_` | 최대 80자, 첫 글자 숫자 불가, 공백 및 기타 특수문자 불가 |
| `인스턴스 타입 선택` | 사전 정의된 스펙 선택 | 선택한 타입의 `vCPU`, `메모리`, 최대 NIC 수가 자동 적용됩니다. |
| `직접 입력` | `vCPU`와 `메모리` 직접 입력 | 필요한 스펙을 직접 지정합니다. |
| `vCPU 상세 설정` | 기본값 사용 또는 옵션 지정 | 적용된 `vCPU = CPU 코어당 스레드 수 × 코어 수`입니다. |

| 필드 | 입력 기준 | 비고 |
| --- | --- | --- |
| `vCPU` | 1 이상의 정수 | 실행 중 편집에서는 현재 값보다 줄일 수 없습니다. |
| `메모리` | 크기 입력 + 단위 선택 | 실행 중 증설은 최소 1GB, 512MB 단위입니다. |

### 2. 설치 유형 선택

| 설치 유형 | 사용할 때 | 알아둘 점 |
| --- | --- | --- |
| `OS 이미지` | 표준 이미지로 빠르게 인스턴스를 만들 때 사용합니다. | Cloud-init 기반 사용자 계정과 초기 설정을 함께 적용할 수 있습니다. 생성 후 이미지 변경은 불가합니다. |
| `ISO 파일` | OS를 직접 설치하거나 골든 이미지를 만들기 위한 초기 VM을 만들 때 사용합니다. | 설치 자유도는 높지만 제한사항이 많습니다. 생성 후 ISO 변경은 불가합니다. |

> 중요
>
> `ISO 파일` 설치 유형은 일반적인 사용자 워크로드보다 OS 설치 후 골든 이미지를 만드는 초기 작업에 적합합니다. 일반적인 인스턴스 생성은 `OS 이미지` 사용을 권장합니다.
>
> `ISO 파일` 설치 유형의 주요 제한사항은 다음과 같습니다.
>
> - `사용자 계정` 섹션을 사용할 수 없습니다.
> - `고급 설정` 전체를 사용할 수 없습니다.
> - 실행 중에는 `편집`할 수 없습니다.
> - 편집 화면에서 `ISO 파일`을 변경할 수 없습니다.
> - `CEPH` 스토리지 도메인을 지원하지 않습니다.

> 참고
>
> Windows를 `ISO 파일`로 설치할 때 부팅 장치 선택 화면이 보이면 `QEMU DVD-ROM` 항목을 선택합니다. Windows 11의 네트워크 요구 사항 우회와 VirtIO 드라이버 설치 예시는 [윈도우 가상머신 ISO 생성 및 콘솔 확인](../../../user-guide/examples-and-labs/04-create-windows-instance-from-iso-and-open-console.md)을 확인합니다.

### 3. 부팅 디스크 크기 설정

| 구분 | 입력 기준 | 제한 |
| --- | --- | --- |
| `OS 이미지` | 선택한 이미지 크기가 기본값으로 입력됩니다. | 이미지 크기보다 작게 설정할 수 없습니다. |
| `ISO 파일` | OS를 설치할 빈 부팅 디스크 크기를 입력합니다. | 1 이상의 정수만 입력할 수 있습니다. 소수점은 입력할 수 없습니다. |

`ISO 파일` 설치 유형에서는 아래 기준을 먼저 참고해 부팅 디스크를 설정합니다.

| 운영체제 계열 | 최소값 | CloudiA 기본값 |
| --- | --- | --- |
| `Unknown` | 4 GB | 10 GB |
| `RHEL 계열` | 10 GiB | 15 GiB |
| `Ubuntu` | 5 GB | 15 GB |
| `Debian` | 4 GB | 8 GB |
| `Windows 10` | 32 GB | 64 GB |
| `Windows 11` | 64 GB | 80 GB |
| `Windows Server` | 32 GB | 64 GB |

> 참고
>
> 운영체제별 상세 기준은 [부록 A. ISO 기반 부팅 디스크 기준](#appendix-a-boot-disk-criteria)을 확인합니다.

### 4. 네트워크 인터페이스 구성

- 인스턴스에는 최소 1개의 네트워크 인터페이스가 필요합니다.
- 인스턴스는 하나의 `VPC`에 속합니다.
- 첫 번째 NIC가 기본 네트워크 인터페이스가 됩니다.

| 항목 | 입력 방법 | 비고 |
| --- | --- | --- |
| `서브넷` | 선택한 VPC에 속한 서브넷을 선택합니다. | NIC마다 1개씩 선택합니다. |
| `IP 주소 - 자동 할당` | 사용 가능한 IPv4를 자동으로 할당합니다. | 권장 기본값입니다. |
| `IP 주소 - 수동 할당` | 선택한 서브넷의 CIDR 안에서 IPv4를 직접 입력합니다. | 서브넷 선택 후 사용할 수 있습니다. |
| `보안그룹` | NIC에 적용할 보안그룹을 선택합니다. | NIC당 최대 5개를 선택합니다. |

> 참고
>
> NIC가 여러 개여도 게스트 OS에는 기본 네트워크 인터페이스의 DNS만 자동으로 적용됩니다. 추가 NIC에 필요한 DNS는 게스트 OS에서 직접 설정합니다.

### 5. 사용자 계정 설정

`사용자 계정`은 `OS 이미지` 설치 유형에서만 사용합니다. `ISO 파일` 설치 유형에서는 이 섹션이 비활성화됩니다.

| 항목 | 입력 기준 | 비고 |
| --- | --- | --- |
| `사용자 이름` | 4~20자, 첫 글자 영문 소문자, 소문자/숫자/`_`/`-` | 예약어는 사용할 수 없습니다. |
| `비밀번호` | 4~20자, 허용된 영문/숫자/특수문자 | `SSH 키`를 선택하지 않으면 필요합니다. |
| `SSH 키` | 프로젝트에 등록된 SSH 키 선택 | 최대 5개까지 선택합니다. `비밀번호`를 사용하지 않으면 1개 이상 선택합니다. |

### 6. 선택 설정 검토

필요한 경우 아래 항목을 함께 설정합니다.

| 항목 | 설명 | 비고 |
| --- | --- | --- |
| `블록` | 프로젝트의 블록 자원을 인스턴스에 연결합니다. | 운영 데이터와 OS 디스크를 분리할 때 사용합니다. |
| `파일 시스템` | 파일 시스템 자원을 선택하고 마운트 포인트를 지정합니다. | `NFS`, `virtiofs` 같은 공유 저장소 연결에 사용합니다. |
| `AI 가속기` | AI 가속기를 연결합니다. | 현재 인스턴스 화면에서는 `NPU` 기준으로 제공합니다. |
| `고급 설정` | 필요하면 `보안 유형`, `최대 vCPU`, `최대 메모리`, `가상 CPU 피닝`을 설정합니다. | 자세한 의미와 운영 기준은 `운영 참고`를 확인합니다. |
| `사용자 스크립트` | 부팅 시 자동으로 실행할 작업을 등록합니다. | Cloud-init이 동작하는 게스트에서 사용합니다. |

> 참고
>
> `NFS` 파일 시스템이 연결된 경우에는 일반적인 경우보다 종료 시간이 길어질 수 있습니다.

### 7. 사용자 스크립트 입력

반복되는 초기 설정이 있으면 `사용자 스크립트`로 자동화합니다.

| 항목 | 설명 |
| --- | --- |
| `실행 순서` | 여러 스크립트를 등록한 경우 실행 순서를 지정합니다. |
| `파일 이름` | 스크립트 파일 이름을 입력합니다. |
| `단계` | 실행 시점을 선택합니다. |
| `내용` | 스크립트 본문을 입력합니다. |

| 단계 | 실행 의미 |
| --- | --- |
| `PER_BOOT` | 부팅할 때마다 실행됩니다. 재부팅하면 다시 실행됩니다. |
| `PER_INSTANCE` | 같은 인스턴스에서는 보통 다시 실행되지 않습니다. 다만 CloudiA에서 `사용자 스크립트` 변경, `SSH 키` 변경, `NIC` 추가·삭제처럼 cloud-init 재적용이 필요한 설정 변경이 발생하면 다시 실행될 수 있습니다. 이 과정에서 내부적으로 datasource의 instance-id가 달라질 수 있습니다. |
| `PER_ONCE` | 한 번만 실행됩니다. 재실행 방법은 `cloud-init clean` 후 재부팅입니다. |

> 참고
>
> 사용자 스크립트는 게스트 OS 안에서 실행됩니다. 패키지 관리자와 서비스 이름은 게스트 OS에 맞게 작성합니다.

## 인스턴스 편집

### 편집을 시작하기

- 목록에서 인스턴스 1개를 선택합니다.
- `편집`을 클릭합니다.
- 편집 화면은 생성 화면과 같은 구성을 사용합니다.
- 생성 후 변경할 수 없는 항목은 잠겨 있습니다.

> 중요
>
> `수동 생성`이 아닌 인스턴스는 인스턴스 메뉴에서 `편집` 버튼이 비활성화됩니다. 해당 인스턴스는 상위 서비스 화면에서 관리합니다.

### 종료 상태에서 편집

`종료됨` 또는 `정지됨` 상태에서는 편집 화면에 들어가 수정 가능한 항목을 저장할 수 있습니다.

- 생성 시에만 결정되는 항목은 계속 잠겨 있습니다.
- 대표적으로 `설치 유형`, `OS 이미지`, `ISO 파일`은 생성 후 변경할 수 없습니다.

### 실행 중 편집

실행 중 편집은 라이브 업데이트로 처리합니다. 화면 상단에 경고 배너가 표시될 수 있습니다.

#### 실행 중에도 가능한 작업

- `vCPU` 늘리기
- `메모리` 늘리기
- 네트워크 인터페이스 추가, 편집, 삭제. 단, 기본 NIC는 제외합니다.
- 추가 블록 디스크 연결 또는 제거

#### 실행 중에는 제한되는 작업

- `인스턴스 이름`
- `인스턴스 타입 선택`과 `직접 입력` 전환
- 하이퍼스레딩 변경
- `설치 유형`, `OS 이미지`, `ISO 파일`, `부팅 디스크 크기`
- `VPC` 변경
- 기본 NIC 편집 또는 삭제
- `사용자 계정`
- `보안 유형`, `최대 vCPU`, `최대 메모리`
- `파일 시스템`
- `AI 가속기`
- `가상 CPU 피닝`
- `사용자 스크립트` 등록

> 중요
>
> `ISO 파일` 설치 유형의 인스턴스는 `실행 중` 상태에서 편집할 수 없습니다.

> 참고
>
> 실행 중 편집에서는 현재 값보다 작은 `vCPU` 또는 `메모리`로 줄일 수 없습니다.
>
> - `vCPU`: 현재 값 이상, 최대 `vCPU` 이하
> - `메모리`: 현재 값 이상, 최대 `메모리` 이하
>
> 보안 유형에 따라 라이브 업데이트가 제한될 수 있습니다.

> 주의
>
> 실행 중에 NIC나 블록 디스크를 추가해도 게스트 OS 내부 설정이 자동으로 끝나지 않을 수 있습니다. 인터페이스, IP, 라우팅, DNS, 디스크 인식 상태는 게스트 OS에서 직접 확인합니다. 자세한 점검 예시는 [부록 B. 다중 NIC 운영 참고](#appendix-b-multi-nic-ops)와 [부록 D. 추가 블록 디스크와 파일 시스템 확인 명령](#appendix-d-block-disk-and-filesystem-checks)을 확인합니다.

### 상세 화면에 들어가기

- 목록에서 인스턴스 이름을 클릭합니다.
- `생성 중`, `삭제 중`, `에러` 상태에서는 상세 화면에 들어갈 수 없습니다.

### 화면에서 확인할 수 있는 정보

| 구분 | 확인할 수 있는 내용 |

| `연결된 리소스` | 파일 시스템, 플로팅 IP, 선호도 그룹, SSH 키 |
| `모니터링 정보` | CPU, 메모리, 디스크, 네트워크 지표 |

### 네트워크 인터페이스 탭

| 항목 | 설명 |
| --- | --- |
| `서브넷` | NIC가 연결된 서브넷 |
| `보안그룹` | NIC에 적용된 보안그룹 |
| `IP 주소` | NIC의 IPv4 주소 |
| `기본 게이트웨이` | 해당 NIC의 게이트웨이 |
| `DNS 서버` | 화면에 표시되는 DNS 정보 |
| `기본 인터페이스` | 기본 NIC 여부 |
| `MAC 주소` | NIC의 MAC 주소 |

> 참고
>
> NIC가 여러 개여도 게스트 OS에는 기본 NIC의 DNS만 자동으로 적용됩니다. 추가 NIC에 필요한 DNS는 게스트 OS에서 직접 설정합니다.

### 디스크 탭

| 항목 | 설명 |
| --- | --- |
| `부팅 순서` | 부팅 시도 순서 |
| `타입` | 디스크 유형 |
| `장치 레이블` | 게스트 OS에 연결되는 디스크 레이블 |
| `버스 타입` | 디스크 버스 종류 |
| `크기` | 디스크 크기 |
| `소스 타입 / 소스 이름` | 디스크 원본 정보 |

### 디스크 부팅 순서 변경

디스크가 2개 이상 연결된 인스턴스는 부팅 순서를 변경할 수 있습니다.

1. 인스턴스 상세 화면으로 이동합니다.
2. `인스턴스 구성 요소 > 디스크`를 선택합니다.
3. 인스턴스가 `종료됨` 상태인지 확인합니다.
4. `부팅 순서 변경`을 클릭합니다.
5. 장치를 드래그해 순서를 바꾸고, 필요하면 부팅 포함 여부를 조정합니다.
6. `저장`을 클릭합니다.

> 주의
>
> 부팅 불가능한 디스크를 가장 높은 우선순위로 두면 게스트 OS가 정상 부팅되지 않을 수 있습니다. 운영체제가 설치된 부팅 디스크를 상단에 둡니다.

## 인스턴스 스냅샷

인스턴스 스냅샷은 특정 시점의 디스크 상태를 저장하고 이후 롤백에 사용할 수 있는 기능입니다.

### 스냅샷으로 할 수 있는 작업

| 작업 | 수행 위치 | 상태 조건 |
| --- | --- | --- |
| `스냅샷 생성` | 목록의 `더보기 > 스냅샷 생성` | `실행 중` 또는 `종료됨` |
| `스냅샷 편집` | 상세 화면 `인스턴스 구성 요소 > 스냅샷` | 스냅샷 1개 선택 |
| `스냅샷 삭제` | 상세 화면 `인스턴스 구성 요소 > 스냅샷` | 스냅샷 1개 선택 |
| `스냅샷 롤백` | 상세 화면 `인스턴스 구성 요소 > 스냅샷` | 인스턴스가 `종료됨` 상태 |

### 스냅샷 생성

1. 목록에서 인스턴스 1개를 선택합니다.
2. `더보기 > 스냅샷 생성`을 클릭합니다.
3. 스냅샷 이름과 설명을 입력합니다.
4. `생성`을 클릭합니다.

### 스냅샷 편집

1. 인스턴스 상세 화면에서 `인스턴스 구성 요소 > 스냅샷`을 선택합니다.
2. 수정할 스냅샷 1개를 선택합니다.
3. `편집`을 클릭합니다.
4. 이름과 설명을 수정합니다.

### 스냅샷 삭제

1. 인스턴스 상세 화면에서 `인스턴스 구성 요소 > 스냅샷`을 선택합니다.
2. 삭제할 스냅샷 1개를 선택합니다.
3. `삭제`를 클릭합니다.

### 스냅샷 롤백

1. 인스턴스가 `종료됨` 상태인지 확인합니다.
2. `인스턴스 구성 요소 > 스냅샷`을 선택합니다.
3. 롤백할 스냅샷 1개를 선택합니다.
4. `롤백`을 클릭합니다.

> 중요
>
> 실행 중 스냅샷의 정합성이 중요하면 `QEMU Guest Agent` 설치 여부를 먼저 확인합니다. `ISO 파일` 설치 유형뿐 아니라 Linux 게스트 기본 이미지에도 게스트 에이전트가 없을 수 있습니다. 설치 예시는 [부록 C. 게스트 에이전트와 Windows 참고](#appendix-c-guest-agent-and-windows)를 확인합니다.

### 상위 서비스 인스턴스 관리

아래와 같은 인스턴스는 인스턴스 메뉴에서 상태와 상세 정보만 우선 확인합니다.

### 인스턴스 실행이 실패할 때 확인할 항목

인스턴스를 `실행`할 때 배치 가능한 물리 호스트는 시스템이 자동으로 선택합니다. 아래 조건을 만족하는 호스트가 없으면 실행이 실패할 수 있습니다.

| 확인 항목 | 설명 |
| --- | --- |
| `인스턴스 구성` | 호스트가 요청한 `vCPU`와 `메모리`를 수용할 수 있어야 합니다. |
| `설치 유형` | 호스트가 필요한 OS 이미지 또는 ISO 파일을 제공할 수 있어야 합니다. |
| `스토리지` | 연결된 블록, 파일 시스템 등 필요한 스토리지 자원을 제공할 수 있어야 합니다. |
| `AI 가속기` | 요청한 NPU 장치 유형이나 카드 수를 충족할 수 있어야 합니다. |

| `가상 CPU 피닝` | 지정한 논리 CPU 범위를 수용할 수 있어야 합니다. |
| `선호도 그룹` | 선호 또는 반선호 규칙을 만족해야 합니다. |
| `호스트 상태` | 호스트 연결 상태, 에이전트 상태, 내부 네트워크 상태가 정상이어야 합니다. |

>

### 추가 블록 디스크와 파일 시스템 연결 확인

추가 리소스를 연결한 뒤에는 화면과 게스트 OS를 함께 확인합니다.

#### 추가 블록 디스크

- 상세 화면의 `디스크` 탭에서 `장치 레이블`과 디스크 정보를 확인합니다.
- 게스트 OS에서 실제 디스크가 인식됐는지 확인합니다.
- Linux에서는 장치 이름만 보지 말고 `SERIAL` 값과 `장치 레이블`을 함께 비교합니다.
- Windows에서는 디스크가 `Offline` 또는 `RAW`인지 함께 확인합니다.

#### 파일 시스템

- 상세 화면의 `연결된 리소스 > 파일 시스템`에서 연결 상태를 확인합니다.
- 게스트 OS에서 실제 마운트 여부와 읽기/쓰기 여부를 확인합니다.
- `NFS`는 관련 패키지가 설치되어 있어야 합니다.
- 게스트 OS에 `NFS` 클라이언트 패키지가 없으면 `/etc/fstab`에 항목이 있어도 자동 마운트가 완료되지 않을 수 있습니다.
- 이 경우 패키지를 설치한 뒤 `mount -a`로 기존 `fstab` 항목을 적용합니다.
- `virtiofs`는 마운트 여부뿐 아니라 읽기/쓰기 테스트까지 확인합니다.

> 참고
>
> Linux와 Windows에서 사용하는 상세 명령 예시는 [부록 D. 추가 블록 디스크와 파일 시스템 확인 명령](#appendix-d-block-disk-and-filesystem-checks)을 확인합니다. NFS 공유 볼륨 구성, 패키지 설치, `fstab` 적용, 마운트 확인을 순서대로 따라 하려면 [파일 시스템을 통한 인스턴스 간 공유 볼륨 구성하기](../../../user-guide/examples-and-labs/17-share-volumes-between-instances-with-nfs.md)를 확인합니다.

### 부록 B. 다중 NIC 운영 참고

#### 같은 서브넷, 다른 서브넷, 외부 네트워크

| 통신 구간 | 일반적인 동작 | 함께 확인할 항목 |
| --- | --- | --- |
| `같은 서브넷` | 상대 인스턴스의 사설 IP로 직접 통신합니다. | 보안그룹, 게스트 OS 방화벽, 서비스 리슨 주소 |
| `같은 VPC의 다른 서브넷` | 각 서브넷의 게이트웨이를 통해 통신합니다. | 보안그룹, 게이트웨이, 게스트 OS 라우팅, 네트워크 정책 |
| `다른 VPC 또는 외부 네트워크` | 별도 연결 정책 없이 직접 통신하지 않습니다. | 라우팅, NAT 또는 플로팅 IP, 보안 정책 |

#### 비대칭 라우팅이 발생하는 경우

비대칭 라우팅은 요청이 들어온 인터페이스와 응답이 나가는 인터페이스가 다른 경우입니다. 다중 NIC 환경에서는 다음 조건에서 자주 발생합니다.

- 여러 NIC에 기본 게이트웨이가 동시에 설정된 경우
- 특정 NIC의 IP로 수신한 요청에 대해 다른 NIC의 기본 경로가 선택된 경우
- 소스 IP 기준 라우팅이 필요한데 기본 라우팅 테이블만 사용하는 경우
- Linux의 `rp_filter`가 strict 모드로 설정된 경우

#### 점검 항목

다중 NIC에서 통신이 되지 않으면 Linux 게스트에서 다음 항목을 함께 점검합니다.

1. 인터페이스별 IP와 라우팅
2. `ip rule` 기반 정책 기반 라우팅(PBR)
3. `rp_filter` 설정
4. 기본 게이트웨이 중복 여부

자주 사용하는 명령은 다음과 같습니다.

```bash
ip addr
ip route
ip route show table all
ip rule

sysctl net.ipv4.conf.all.rp_filter
sysctl net.ipv4.conf.default.rp_filter
sysctl net.ipv4.conf.<인터페이스명>.rp_filter
```

#### 수동 IP 설정 예시(Linux)

실행 중 편집으로 NIC를 추가한 뒤 게스트 OS에서 새 인터페이스를 직접 구성해야 할 수 있습니다.

```bash
IFACE=eth1
IPADDR=10.0.0.66/24
GW=10.0.0.1
DNS1=10.0.0.2

nmcli con add type ethernet ifname "$IFACE" con-name "$IFACE" ip4 "$IPADDR" gw4 "$GW"
nmcli con modify "$IFACE" ipv4.dns "$DNS1" ipv4.method manual
nmcli con up "$IFACE"
```

> 참고
>
> 기본 NIC가 아닌 인터페이스에 게이트웨이를 추가하면 별도의 default route가 생길 수 있습니다. 다중 NIC 환경에서는 경로 우선순위와 정책 기반 라우팅을 함께 확인합니다.

### 부록 D. 추가 블록 디스크와 파일 시스템 확인 명령

#### Linux에서 추가 블록 디스크 확인

1. 상세 화면에서 추가 블록 디스크의 `장치 레이블`을 확인합니다.
2. 게스트 OS에서 다음 명령을 실행합니다.

```bash
lsblk -o NAME,SERIAL
```

3. `SERIAL` 값과 상세 화면의 `장치 레이블` 숫자를 함께 비교합니다.

> 참고
>
> 디스크가 여러 개 연결된 경우에는 `/dev/vdb`, `/dev/vdc` 같은 장치 이름만으로 식별하지 말고 `SERIAL` 값과 `장치 레이블`을 함께 확인합니다.

#### Windows에서 추가 블록 디스크 확인

```powershell
Get-Disk
Get-Partition
Get-Volume
```

새 디스크가 `Offline` 또는 `RAW`로 보이면 초기화, 파티션 생성, 포맷, 드라이브 문자 할당을 순서대로 진행합니다.

#### Linux에서 NFS 준비 사항 확인

| 게스트 OS | 확인할 패키지 | 확인 명령 |
| --- | --- | --- |
| `Rocky Linux`, `RHEL` 계열 | `nfs-utils` | `rpm -q nfs-utils` |
| `Ubuntu` | `nfs-common` | `dpkg -l nfs-common` |

필요한 패키지가 없으면 다음 명령으로 설치합니다.

```bash
# Rocky Linux / RHEL 계열
sudo dnf install -y nfs-utils

# Ubuntu
sudo apt-get update
sudo apt-get install -y nfs-common
```

패키지를 설치한 뒤에는 `/etc/fstab`에 이미 기록된 NFS 항목을 적용합니다.

```bash
sudo mount -a
```

#### virtiofs 확인 예시

```bash
mountpoint /mnt/virtiofs
df -h /mnt/virtiofs
sudo sh -c 'echo "Hello ---" > /mnt/virtiofs/test-file.txt'
cat /mnt/virtiofs/test-file.txt
ls -la /mnt/virtiofs/
```

#### NFS 확인 예시

```bash
sudo cat /etc/fstab
sudo mount -a
findmnt
df -hT
mount | grep nfs
```

### 부록 E. 외부 접속 또는 내부 통신 점검

통신이 되지 않으면 다음 순서로 확인합니다.

1. 인스턴스 상태가 `실행 중`인지 확인합니다.
2. 필요한 NIC에 올바른 IP가 설정됐는지 확인합니다.
3. 보안그룹 인바운드와 아웃바운드 규칙을 확인합니다.
4. 게스트 OS 방화벽과 서비스 리슨 주소를 확인합니다.
5. 플로팅 IP가 필요한 구조라면 올바른 인터페이스에 연결됐는지 확인합니다.
6. 다중 NIC 환경이라면 라우팅, 정책 기반 라우팅, `rp_filter`, 기본 게이트웨이 우선순위를 함께 확인합니다.
