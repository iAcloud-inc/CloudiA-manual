# 인스턴스

인스턴스는 가상머신을 생성하고 실행 상태를 관리하며, 네트워크·디스크·파일 시스템·스냅샷 같은 연관 리소스를 함께 운영하는 메뉴입니다. 이 문서는 인스턴스 목록, 생성·편집, 상세, 스냅샷, 다중 NIC 운영 시 주의사항을 사용자 매뉴얼 형식으로 설명합니다.

## 화면 개요

- **목록 화면**: 인스턴스 현황을 확인하고 생성, 편집, 전원 제어, 삭제, 스냅샷 생성 같은 작업을 시작합니다.
- **생성 화면**: 이름, 스펙, 설치 유형, 네트워크, 사용자 계정, 고급 설정, 사용자 스크립트를 입력합니다.
- **편집 화면**: 생성 화면과 동일한 구성을 사용하지만, 생성 후 변경할 수 없는 항목은 잠겨 있으며 실행 중에는 일부 항목만 수정할 수 있습니다.
- **상세 화면**: 인스턴스 상태와 연결된 리소스를 확인하고, 디스크 부팅 순서 변경, 스냅샷 관리, AI 가속기 확인, 가상 CPU 피닝 확인 같은 후속 작업을 수행합니다.

## 인스턴스 목록

인스턴스 목록 화면은 프로젝트에 생성된 인스턴스를 조회하고, 상태와 생성 방식을 기준으로 관리 대상을 구분하는 화면입니다. 목록에서는 **생성 방식**, **상태**, **IP 주소**를 우선 확인합니다.

### 인스턴스 상태

인스턴스 상태는 현재 전원 상태와 관리 가능 범위를 나타냅니다.

| 상태                | 의미                                          | 상세 진입 | 콘솔 |
| ------------------- | --------------------------------------------- | --------- | ---- |
| **생성 중**         | 인스턴스 생성 작업이 진행 중입니다.           | 불가      | 불가 |
| **부팅 중**         | 전원 인가 후 게스트 OS가 올라오는 중입니다.   | 가능      | 가능 |
| **실행 중**         | 게스트 OS가 동작 중입니다.                    | 가능      | 가능 |
| **재부팅 중**       | 재부팅 작업이 진행 중입니다.                  | 가능      | 가능 |
| **종료 중**         | 종료 작업이 진행 중입니다.                    | 가능      | 가능 |
| **종료됨**          | 전원이 꺼진 상태입니다. `실행`할 수 있습니다. | 가능      | 불가 |
| **정지됨**          | 전원이 꺼진 상태입니다. `실행`할 수 있습니다. | 가능      | 불가 |
| **일시 중지**       | 실행 상태에서 일시 중지된 상태입니다.         | 가능      | 가능 |
| **마이그레이션 중** | 호스트 간 이동 작업이 진행 중입니다.          | 가능      | 가능 |
| **삭제 중**         | 삭제 작업이 진행 중입니다.                    | 불가      | 불가 |
| **에러**            | 생성 또는 운영 중 오류가 발생한 상태입니다.   | 불가      | 불가 |

> 참고
>
> 이름 클릭으로 상세 화면에 진입할 수 없는 상태는 `생성 중`, `삭제 중`, `에러`입니다.

### 주요 작업

- `생성`: 새 인스턴스를 만듭니다. 생성 버튼은 항상 표시됩니다.
- `편집`: 선택한 **1개** 인스턴스를 수정합니다. `종료됨`, `정지됨`, `실행 중` 상태에서 사용할 수 있으며, `ISO 파일` 설치 유형의 인스턴스는 `실행 중`일 때 편집할 수 없습니다.
- `삭제`: 선택한 **1개** 인스턴스에 대해 삭제 확인 창을 엽니다.
  - 목록 버튼은 `생성 중`, `실행 중`, `삭제 중`, `부팅 중`, `재부팅 중`, `종료 중` 상태에서는 비활성화됩니다.
  - 삭제 확인 창이 열리더라도, 실제 삭제는 **수동 생성 인스턴스**이고 **연결된 인스턴스 스냅샷이 0개**일 때만 실행할 수 있습니다.
- `실행`: 전원이 꺼진 인스턴스를 켭니다. `종료됨`, `정지됨` 상태이면서 **수동 생성** 인스턴스일 때 활성화됩니다.
- `종료`: 실행 중인 인스턴스를 정상 종료합니다. `실행 중`, `일시 중지` 상태이면서 **수동 생성** 인스턴스일 때 활성화됩니다.
- `재부팅`: 실행 중인 인스턴스를 다시 시작합니다. `실행 중` 상태이면서 **수동 생성** 인스턴스일 때 활성화됩니다. 보안 유형에 따라 재부팅이 제한될 수 있습니다.
- `더보기 > 스냅샷 생성`: 인스턴스 상태를 스냅샷으로 저장합니다. `실행 중` 또는 `종료됨` 상태에서 사용할 수 있으며, 진행 중인 작업이 있는 경우에는 비활성화될 수 있습니다.

> 참고:
> 목록에서는 좌측 **라디오 버튼**으로 인스턴스를 선택합니다.
> 한 번에 **1개 인스턴스만 선택**할 수 있으며, 상단 작업도 모두 단일 선택 기준으로 동작합니다.

### 목록 컬럼

| 컬럼          | 설명                                                         |
| ------------- | ------------------------------------------------------------ |
| **이름**      | 인스턴스 표시 이름입니다. 클릭하면 상세 화면으로 이동합니다. |
| **ID**        | 인스턴스 고유 ID입니다.                                      |
| **생성 방식** | 인스턴스가 생성된 경로입니다. `수동 생성`, `오토 스케일링 그룹`, `쿠버네티스`, `파일 시스템`, `Gen:iA` 등으로 표시됩니다. |
| **IP 주소**   | NIC에 할당된 IPv4 주소를 표시합니다. NIC가 여러 개이면 쉼표로 구분해 함께 표시합니다. **가장 앞에 표시되는 IPv4 주소가 기본 네트워크 인터페이스의 IPv4 주소입니다.** |
| **상태**      | 현재 인스턴스 상태를 색상 점과 텍스트로 표시합니다.          |
| **vCPU**      | 인스턴스에 할당된 vCPU 개수입니다. vCPU 개수는 **CPU 코어당 스레드 수 × 코어 수**입니다. |
| **메모리**    | 인스턴스에 할당된 메모리 크기를 표시합니다. |
| **생성 일시** | 인스턴스가 생성된 날짜와 시간입니다.                         |

### 생성 방식별 동작 제한

인스턴스의 실제 동작 가능 범위는 상태뿐 아니라 **생성 방식**에도 영향을 받습니다.

| 생성 방식              | 실행(종료 상태) | 편집 | 종료 | 재부팅 | 삭제   | 관리 위치                   |
| ---------------------- | --------------- | ---- | ---- | ------ | ------ | --------------------------- |
| **수동 생성**          | 가능            | 가능 | 가능 | 가능   | 조건부 | 인스턴스 화면에서 직접 관리 |
| **오토 스케일링 그룹** | 가능            | 제한 | 제한 | 제한   | 제한   | 오토 스케일링 그룹 화면     |
| **쿠버네티스**         | 가능            | 제한 | 제한 | 제한   | 제한   | 쿠버네티스 화면             |
| **파일 시스템**        | 가능            | 제한 | 제한 | 제한   | 제한   | 파일 시스템 연계 기능 화면  |
| **Gen:iA**             | 가능            | 제한 | 제한 | 제한   | 제한   | Gen:iA 연계 화면            |

> 중요
>
> `수동 생성`이 아닌 인스턴스는 **종료 상태일 때 `실행`은 가능할 수 있지만**, `삭제`, `편집`, `종료`, `재부팅`은 인스턴스 화면에서 제한됩니다. 해당 인스턴스는 상위 서비스 화면에서 관리하십시오.

### 삭제 가능 조건

삭제 버튼의 활성화 여부와 실제 삭제 가능 조건은 다릅니다.

| 항목                        | 조건                                                         |
| --------------------------- | ------------------------------------------------------------ |
| **삭제 버튼 비활성화 상태** | `생성 중`, `실행 중`, `삭제 중`, `부팅 중`, `재부팅 중`, `종료 중` |
| **실제 삭제 가능 조건**     | `수동 생성` 인스턴스이고, 연결된 인스턴스 스냅샷 개수가 `0개` |

### 검색과 목록 동작

- **검색 조건**: `이름`
- **검색어 입력**: 인스턴스 표시 이름
- **정렬**: 대부분의 컬럼은 헤더에서 정렬할 수 있습니다. 단, **IP 주소 컬럼은 정렬되지 않습니다.**
- **기본 정렬**: 생성 일시 기준
- **페이지 크기**: 페이지당 10개

## 인스턴스 생성 및 편집

인스턴스 생성과 편집은 동일한 화면 구성을 사용합니다. 다만 편집 화면에서는 생성 후 변경할 수 없는 항목이 잠기며, 인스턴스가 실행 중이면 라이브 업데이트가 허용된 항목만 수정할 수 있습니다.

### 생성 전 준비 사항

이 항목은 생성 절차가 아니라, 인스턴스 생성 전에 확인할 **사전 준비 조건**입니다.

- 인스턴스가 연결될 **VPC**가 미리 생성되어 있어야 합니다.
- 각 NIC에 연결할 **서브넷**이 선택한 VPC 안에 미리 생성되어 있어야 합니다.
- 접속 방식을 **비밀번호**, **SSH 키**, 또는 둘 다 사용할지 결정합니다.
- 추가 블록 디스크, 파일 시스템, AI 가속기(NPU)가 필요한지 확인합니다.
- 설치 방식을 `OS 이미지`로 할지 `ISO 파일`로 할지 결정합니다.

### 실행 중 편집(라이브 업데이트)

라이브 업데이트는 실행 중인 인스턴스에 허용된 변경을 반영하는 방식입니다. 이 중 NIC 추가나 블록 디바이스 추가처럼 실행 중인 인스턴스에 장치를 연결하는 작업을 핫 플러그라고 합니다.

인스턴스가 `실행 중`이면 화면 상단에 경고 배너가 표시됩니다.

**실행 중에도 가능한 작업**

- vCPU 변경
- 메모리 증설
- 네트워크 인터페이스 추가/편집/삭제(기본 NIC 제외)
- 블록 디바이스 추가/삭제

**실행 중에는 제한되는 작업**

- 인스턴스 이름
- 인스턴스 타입 선택/직접 입력 전환
- 하이퍼스레딩 변경
- 설치 유형, OS 이미지, ISO 파일, 부팅 디스크 크기
- VPC 변경
- 기본 네트워크 인터페이스 편집/삭제
- 사용자 계정
- 보안 유형, 확장 한도
- 파일 시스템
- AI 가속기
- 가상 CPU 피닝 추가
- 사용자 스크립트 등록

**게스트 OS 내부 조치가 필요한 작업**

- 실행 중 네트워크 인터페이스를 추가하면 화면에는 NIC와 IP가 반영되더라도, **게스트 OS 내부 네트워크 설정은 자동으로 완료되지 않을 수 있습니다.** 운영체제 내부에서 인터페이스, IP, 라우팅, DNS를 확인해야 합니다.

## 1. 기본 정보

### 1.1 인스턴스 이름

- 인스턴스를 식별할 이름을 입력합니다.
- 이름은 목록과 상세 화면에서 공통으로 사용하는 표시 이름입니다.

| 필드              | 허용 값                        | 검증 규칙                                                | 비고 |
| ----------------- | ------------------------------ | -------------------------------------------------------- | ---- |
| **인스턴스 이름** | 영문 대/소문자, 숫자, `-`, `_` | 최대 80자, 첫 글자 숫자 불가, 공백 및 기타 특수문자 불가 | 필수 |

### 1.2 인스턴스 구성

인스턴스 스펙은 **인스턴스 타입 선택** 또는 **직접 입력**으로 설정합니다.

#### 인스턴스 타입 선택

- 사전에 정의된 스펙을 선택합니다.
- 선택한 타입의 vCPU, 메모리, 최대 NIC 개수가 자동으로 적용됩니다.
- 모달에서 `이름`, `vCPU`, `메모리`, `GPU`, `연산 성능`을 확인할 수 있습니다.

#### 직접 입력

- `vCPU`와 `메모리`를 직접 입력합니다.
- 직접 입력 모드에서는 기본적으로 **최대 NIC 100개 기준**으로 동작합니다.
- 실행 중 편집에서는 현재 값보다 작게 줄일 수 없고, 최대 vCPU/메모리 범위 내에서만 늘릴 수 있습니다.

#### vCPU 상세 설정

상세 설정에서 vCPU 내부 구성을 조정할 수 있습니다.

| 항목                     | 설명                                                      |
| ------------------------ | --------------------------------------------------------- |
| **기본 vCPU 사용**       | 현재 선택한 vCPU 값을 기준으로 기본 구성을 사용합니다.    |
| **옵션 지정**            | 하이퍼스레딩 사용 여부와 코어 수를 직접 지정합니다.       |
| **CPU 코어당 스레드 수** | 하이퍼스레딩 사용 시 일반적으로 2, 사용 안 함 시 1입니다. |
| **코어 수**              | 가상 CPU 코어 개수입니다.                                 |
| **적용된 vCPU**          | **CPU 코어당 스레드 수 × 코어 수**로 계산합니다.          |

| 필드       | 허용 값               | 검증 규칙                                                    | 비고                                   |
| ---------- | --------------------- | ------------------------------------------------------------ | -------------------------------------- |

## 2. 네트워크 인터페이스 구성

인스턴스에는 최소 1개의 네트워크 인터페이스가 필요합니다. 인스턴스는 하나의 VPC에 속하며, 각 NIC는 선택한 VPC에 속한 서브넷 중 하나에 연결합니다. 이 섹션에서는 NIC별 IP 주소와 보안그룹을 함께 설정합니다.

### 2.1 기본 규칙

- 첫 번째로 추가한 NIC가 **기본 네트워크 인터페이스**가 됩니다.
- 목록의 IP 주소 컬럼에서 **가장 앞의 IPv4 주소가 기본 네트워크 인터페이스의 IPv4 주소**입니다.
- NIC가 여러 개여도 **게스트 OS에는 기본 네트워크 인터페이스의 DNS만 자동 설정**됩니다.
- NIC마다 게이트웨이가 있으면, 게스트 OS 또는 네트워크 관리 도구가 **NIC별 라우트**를 생성할 수 있습니다.
- 실행 중에는 **기본 네트워크 인터페이스를 편집하거나 삭제할 수 없습니다.**

### 2.2 인터페이스 추가·편집

| 항목                    | 설명                                                         | 비고                     |
| ----------------------- | ------------------------------------------------------------ | ------------------------ |
| **서브넷**              | 선택한 VPC에 소속된 서브넷 중, NIC가 연결될 대상 서브넷을 선택합니다. | 최소 1개 NIC 필요        |
| **IP 주소 - 자동 할당** | 선택한 서브넷의 사용 가능한 IPv4를 자동 할당합니다.          |                          |
| **IP 주소 - 수동 할당** | 선택한 서브넷의 CIDR 범위 안에서 IPv4를 직접 입력합니다.     | 서브넷 선택 후 사용 가능 |
| **보안그룹**            | NIC에 적용할 보안그룹을 선택합니다.                          | 최대 5개                 |

### 2.3 같은 서브넷 / 다른 서브넷 통신

같은 VPC 안에서도 **같은 서브넷인지, 다른 서브넷인지**에 따라 통신 경로가 달라집니다.

| 통신 구간                       | 일반적인 동작                                                | 운영자가 확인할 항목                                         |
| ------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **같은 서브넷**                 | 인스턴스는 상대 인스턴스의 사설 IP로 직접 통신합니다. | 보안그룹, 게스트 OS 방화벽, 서비스 리슨 주소                 |
| **같은 VPC의 다른 서브넷**      | 인스턴스는 자신이 속한 서브넷의 게이트웨이를 통해 상대 서브넷과 통신합니다. 요청과 응답이 모두 정상적으로 오가려면 양쪽 경로가 함께 맞아야 합니다. | 보안그룹, 서브넷의 기본 게이트웨이, 게스트 OS 라우팅, 네트워크 정책 |
| **다른 VPC 또는 외부 네트워크** | 기본적으로 별도 연결 정책 없이 직접 통신하지 않습니다. 필요한 경우 외부 노출 또는 별도 네트워크 연결 구성을 추가해야 합니다. | VPC 연결 정책, 라우팅, NAT/플로팅 IP, 보안 정책              |

> 참고
>
> 같은 서브넷이라도 보안그룹이나 게스트 OS 방화벽이 차단하면 통신되지 않습니다. 같은 VPC의 다른 서브넷은 VPC 내부 통신 경로를 사용하더라도, 보안 정책과 게스트 OS 라우팅이 함께 맞아야 합니다.

### 2.4 다중 NIC 운영 시 주의사항

#### 비대칭 라우팅이란

**비대칭 라우팅**은 요청이 들어온 인터페이스와 응답이 나가는 인터페이스가 서로 다른 현상입니다. 예를 들어 `eth1`의 IP로 요청을 받았는데 응답은 `eth0`의 기본 게이트웨이로 나가면, 상대 장비나 게스트 OS의 보안 기능이 패킷을 비정상으로 판단해 통신이 실패할 수 있습니다.

#### 왜 발생하는가

다음 조건에서 비대칭 라우팅이 자주 발생합니다.

- NIC가 여러 개인데 **여러 기본 게이트웨이**가 동시에 설정된 경우
- 특정 NIC의 IP로 수신한 요청에 대해 **다른 NIC의 기본 경로**가 선택된 경우
- 소스 IP 기준 라우팅이 필요한데 기본 라우팅 테이블만 사용하는 경우
- Linux의 `rp_filter`가 strict 모드로 설정되어 있어, 수신 인터페이스와 역방향 경로가 일치하지 않는 패킷을 드롭하는 경우

#### 점검 항목

다중 NIC를 사용할 때 통신이 되지 않으면, Linux 게스트에서 다음 항목을 점검합니다.

1. **인터페이스별 IP와 라우팅**

- `ip addr`
- `ip route`
- `ip route show table all`

2. **정책 기반 라우팅(PBR)**

- `ip rule`
- 소스 IP별로 별도 라우팅 테이블이 필요한지 확인

3. **`rp_filter` 설정**

- `sysctl net.ipv4.conf.all.rp_filter`
- `sysctl net.ipv4.conf.default.rp_filter`
- `sysctl net.ipv4.conf.<인터페이스명>.rp_filter`

4. **기본 게이트웨이 중복 여부**

- 여러 NIC에 기본 게이트웨이를 동시에 두지 않는지 확인
- 필요 없는 NIC는 기본 경로에서 제외하거나, 라우팅 우선순위를 낮춥니다.

> 참고
>
> 다중 NIC 환경에서 통신 문제가 발생하면, **게스트 OS의 인터페이스별 라우팅, `ip rule` 기반 정책 라우팅(PBR), `rp_filter`, 기본 게이트웨이 우선순위**를 함께 점검합니다. 이 점검 항목은 Linux 게스트 기준입니다.

### 2.5 수동 IP 설정 예시(Linux)

실행 중 편집으로 NIC를 추가한 경우, 게스트 OS 내부에서 새 인터페이스를 직접 구성해야 할 수 있습니다.

1. 인스턴스 상세 화면에서 새 인터페이스에 할당된 IP, 게이트웨이, DNS를 확인합니다.
2. 게스트 OS에서 인터페이스 이름을 확인합니다.

```bash
ip link
```

3. 다음 예시처럼 연결을 추가합니다.

```bash
IFACE=eth1
IPADDR=10.0.0.66/24
GW=10.0.0.1
DNS1=10.0.0.2

nmcli con add type ethernet ifname "$IFACE" con-name "$IFACE" ip4 "$IPADDR" gw4 "$GW"
nmcli con modify "$IFACE" ipv4.dns "$DNS1" ipv4.method manual
nmcli con up "$IFACE"
```

> 참고
>
> 기본 NIC가 아닌 인터페이스에 게이트웨이를 추가하면 별도의 default route가 생성될 수 있습니다. 다중 NIC 환경에서는 경로 우선순위와 정책 기반 라우팅 설정을 함께 확인합니다.

## 3. 사용자 계정

`OS 이미지` 설치 유형에서는 Cloud-init으로 사용자 계정과 인증 방식을 설정할 수 있습니다. `ISO 파일` 설치 유형에서는 이 섹션이 비활성화됩니다.

### 3.1 사용자 이름과 인증 방식

| 항목            | 설명                                              |
| --------------- | ------------------------------------------------- |
| **사용자 이름** | 게스트 OS에 생성할 로그인 계정 이름               |
| **비밀번호**    | 콘솔 또는 SSH 비밀번호 인증에 사용                |
| **SSH 키**      | 프로젝트에 등록된 SSH 키를 최대 5개까지 선택 가능 |

### 3.2 검증 규칙

| 필드            | 허용 값                                          | 비고                                      |
| --------------- | ------------------------------------------------ | ----------------------------------------- |
| **사용자 이름** | 4~20자, 첫 글자 영문 소문자, 소문자/숫자/`_`/`-` | 예약어 불가, `OS 이미지`에서 필수         |
| **비밀번호**    | 4~20자, 허용된 영문/숫자/특수문자                | SSH 키가 없으면 필요                      |
| **SSH 키**      | 프로젝트에 등록된 SSH 키 목록                    | 최대 5개, 비밀번호가 없으면 1개 이상 필요 |

## 4. 고급 설정 (선택)

`고급 설정`을 펼치면 인스턴스의 보안/확장 한도, 스토리지, AI 가속기, CPU 피닝을 설정할 수 있습니다. 성능 최적화, 보안 강화, 운영 편의성 확보가 필요한 경우 이 영역을 함께 사용합니다.

> 주의:
> 설치 유형이 `ISO 파일`이면 고급 설정 전체가 비활성화됩니다.

### 4.1 보안 및 확장 설정

보안 요구 사항이 있는 워크로드이거나, 실행 중 증설을 염두에 두고 인스턴스를 만들 때 사용하는 영역입니다.

| 항목            | 설명                                     |
| --------------- | ---------------------------------------- |
| **보안 유형**   | `사용 안 함`, `SEV` |
| **최대 vCPU**   | 실행 중 증설 가능한 vCPU 상한            |
| **최대 메모리** | 실행 중 증설 가능한 메모리 상한          |

### 4.2 스토리지 추가

운영 데이터와 OS 디스크를 분리하거나, 공유 저장소를 연결해야 할 때 사용하는 영역입니다.

| 항목            | 설명                                                         | 비고                                          |
| --------------- | ------------------------------------------------------------ | --------------------------------------------- |
| **블록**        | 프로젝트의 블록 자원을 인스턴스에 연결합니다.                | 실행 중 편집에서는 추가 블록만 추가/삭제 가능 |
| **파일 시스템** | 파일 시스템(NFS, virtiofs) 자원을 선택하고 마운트 포인트를 입력합니다. | 실행 중 편집에서 변경 불가                    |

> 참고
>
> NFS 파일 시스템이 연결된 경우에는 종료 시간이 일반적인 경우보다 길어질 수 있습니다.

### 4.3 AI 가속기

현재 인스턴스 화면의 AI 가속기는 **NPU 기준**으로 제공됩니다.

### 4.4 가상 CPU 피닝

가상 CPU 피닝은 인스턴스의 vCPU를 호스트의 논리 CPU 범위에 고정하는 설정입니다.

| 필드                    | 허용 값       | 검증 규칙                    |
| ----------------------- | ------------- | ---------------------------- |
| **vCPU ID**             | 0 이상의 정수 | 현재 인스턴스 vCPU 개수 미만 |
| **논리 CPU 범위(시작)** | 0 이상의 정수 | 허용 범위 이내               |
| **논리 CPU 범위(끝)**   | 0 이상의 정수 | 시작 값 이상, 중복 범위 불가 |

## 5. 사용자 스크립트

사용자 스크립트는 인스턴스 부팅 시 Cloud-init을 통해 자동 실행되는 작업입니다. 반복되는 초기 설정 작업은 사용자 스크립트로 자동화할 수 있습니다.

### 5.1 입력 항목

| 항목          | 설명                                                |
| ------------- | --------------------------------------------------- |
| **실행 순서** | 여러 스크립트를 등록한 경우 실행 순서를 지정합니다. |
| **파일 이름** | 스크립트 파일 이름                                  |
| **단계**      | 실행 주기                                           |
| **내용**      | 스크립트 본문                                       |

### 5.2 단계별 실행 의미

| 단계             | 실행 의미                                                    |
| ---------------- | ------------------------------------------------------------ |
| **PER_BOOT**     | 부팅할 때마다 실행됩니다. 버전 변경과 무관하게 **재부팅하면 다시 실행**됩니다. |
| **PER_INSTANCE** | 같은 인스턴스면 보통 다시 실행되지 않습니다. 다만 datasource 기준으로 instance-id가 바뀌는 수준의 큰 변경이 있으면 **재실행될 수 있습니다**. cloud-init 버전 변경 자체가 재실행 조건은 아닙니다. |
| **PER_ONCE**     | **한 번만 실행**됩니다. 인스턴스 변경도 재실행을 강제하지 않습니다. 문서상 재실행 방법은 `cloud-init clean` 후 재부팅입니다. |

### 5.3 사용자 스크립트 예시

다음 예시는 Rocky Linux와 Ubuntu에서 공통으로 많이 사용하는 **QEMU Guest Agent 설치 및 활성화** 예시입니다.

**파일 이름 예시**

```text
10-enable-qemu-guest-agent.sh
```

**내용 예시**

```bash
#!/bin/bash
set -euxo pipefail

if command -v dnf >/dev/null 2>&1; then
  dnf install -y qemu-guest-agent
elif command -v apt-get >/dev/null 2>&1; then
  apt-get update
  DEBIAN_FRONTEND=noninteractive apt-get install -y qemu-guest-agent
fi

systemctl enable --now qemu-guest-agent
```

> 참고
>
> 사용자 스크립트는 게스트 OS 안에서 실행됩니다. 패키지 관리자와 서비스 이름은 게스트 OS에 맞게 작성해야 합니다.

## 인스턴스 상세

인스턴스 상세 화면에서는 현재 인스턴스의 상태, 연결된 리소스, 모니터링 정보를 확인합니다.

### 상단 정보

상단 정보 박스에는 다음 정보가 표시됩니다.

- **기본 정보**: 이름, 상태, ID, UUID, VPC, 물리 머신, 생성 일시
- **인스턴스 구성 및 보안**: 인스턴스 타입, vCPU, 메모리, 최대 vCPU, 최대 메모리, 보안 유형
- **프로비저닝 정보**: 생성 방식, 설치 유형, OS 이미지 또는 ISO 파일, 시작 템플릿, 사용자 계정 정보

### 하단 구성

- **인스턴스 구성 요소**: 네트워크 인터페이스, 디스크, 스냅샷, 호스트 장치, AI 가속기, 사용자 스크립트, 가상 CPU 피닝
- **연결된 리소스**: 파일 시스템, 플로팅 IP, 선호도 그룹, SSH 키
- **모니터링 정보**: CPU/메모리, 디스크, 네트워크 지표

### 네트워크 인터페이스 탭에서 확인할 항목

| 항목                | 설명                     |
| ------------------- | ------------------------ |
| **서브넷**          | NIC가 연결된 서브넷      |
| **보안그룹**        | NIC에 적용된 보안그룹    |
| **IP 주소**         | NIC의 IPv4 주소          |
| **기본 게이트웨이** | 해당 NIC의 게이트웨이    |
| **DNS 서버**        | 화면에 표시되는 DNS 정보 |
| **기본 인터페이스** | 기본 NIC 여부            |
| **MAC 주소**        | NIC의 MAC 주소           |

> 중요
>
> NIC가 여러 개여도 **게스트 OS에는 기본 인터페이스의 DNS만 자동으로 적용**됩니다. 추가 NIC에 필요한 DNS는 게스트 OS에서 직접 설정하십시오.

### 디스크 탭에서 확인할 항목

| 항목                      | 설명                               |
| ------------------------- | ---------------------------------- |
| **부팅 순서**             | 부팅 시도 순서                     |
| **타입**                  | 디스크 유형                        |
| **장치 레이블**           | 게스트 OS에 연결되는 디스크 레이블 |
| **버스 타입**             | 디스크 버스 종류                   |
| **크기**                  | 디스크 크기                        |
| **소스 타입 / 소스 이름** | 디스크 원본 정보                   |

### 디스크 부팅 순서 변경

디스크가 2개 이상 연결된 인스턴스는 부팅 시도 순서를 변경할 수 있습니다.

1. 인스턴스 상세 화면으로 이동합니다.
2. `인스턴스 구성 요소 > 디스크`를 선택합니다.
3. 인스턴스가 `종료됨` 상태인지 확인합니다.
4. **부팅 순서 변경**을 클릭합니다.
5. 장치를 드래그하여 순서를 변경하고 필요하면 부팅 포함 여부를 조정합니다.
6. **저장**을 클릭합니다.

> 주의
>
> 부팅 불가능한 디스크 장치를 가장 높은 우선순위로 설정하면 **게스트 OS가 정상 부팅되지 않을 수 있습니다.** 운영체제가 설치된 부팅 디스크를 우선순위 상단에 두십시오.

## 인스턴스 실행 시 호스트 배치 조건

인스턴스를 `실행`할 때 인스턴스가 배치되는 물리 호스트는 시스템이 자동으로 선택합니다.
- 인스턴스 실행 시점의 호스트 자원 상태와 정책에 따라 배치 가능한 호스트가 달라질 수 있습니다.
- 조건을 만족하는 호스트가 없으면 인스턴스를 실행할 수 없습니다.

아래 표는 인스턴스 설정에 따라 호스트 배치가 제한될 수 있는 주요 상황을 정리한 것입니다.

| 인스턴스 설정 | 호스트 배치 제약 요인 |
|---|---|
| **인스턴스 구성 (vCPU, 메모리)** | 호스트가 인스턴스에 설정된 vCPU/메모리를 수용할 여유가 부족한 경우 |
| **설치 유형 (`OS 이미지`, `ISO 파일`)** | 호스트가 해당 설치 유형에 필요한 OS 이미지 또는 ISO 파일을 제공할 수 없는 경우 |
| **스토리지 (블록, 파일 시스템)** | 호스트가 인스턴스에 연결된 스토리지 자원을 제공할 수 없는 경우 |
| **AI 가속기 (NPU 장치, 카드 수)** | 호스트가 요청된 NPU 장치 유형이나 카드 수를 충족할 수 없는 경우 |
| **보안 유형** | 호스트가 인스턴스의 보안 유형을 지원하지 않거나 추가 수용이 어려운 경우 |
| **가상 CPU 피닝** | 호스트가 인스턴스의 가상 CPU 피닝 범위를 지원하지 않는 경우 |
| **선호도 그룹** | 인스턴스가 속한 선호도 그룹의 강제 규칙(선호/반선호)을 만족하는 배치가 어려운 경우 |

> 주의
>
> 호스트 연결 상태, 호스트 에이전트/VDSM 상태, 내부 네트워크 상태 같은 기본 동작 조건도 함께 확인합니다.

> 참고
>
> 인스턴스 설정을 만족하는 호스트 후보가 여러 개인 경우, 내부 정책으로 우선순위를 산정해 최종 호스트를 선택합니다.

> 참고
>
> 자원 여유 판단은 모니터링 화면의 실시간 사용률과 동일하지 않으며, 시스템 내부에서 관리하는 호스트별 배치 자원 기준으로 수행됩니다.

## 인스턴스 스냅샷

인스턴스 스냅샷은 특정 시점의 디스크 상태를 저장하고 이후 롤백에 사용할 수 있는 기능입니다.

### 스냅샷 생성

1. 목록 화면에서 인스턴스를 선택합니다.
2. **더보기 > 스냅샷 생성**을 클릭합니다.
3. 스냅샷 이름과 설명을 입력합니다.
4. **생성**을 클릭합니다.

### 스냅샷 생성 가능 상태

| 상태           | 가능 여부      |
| -------------- | -------------- |
| **실행 중**    | 가능           |
| **종료됨**     | 가능           |
| **그 외 상태** | 불가 또는 제한 |

### 스냅샷 편집

1. 인스턴스 상세 화면에서 `인스턴스 구성 요소 > 스냅샷`을 선택합니다.
2. 수정할 스냅샷 1개를 선택합니다.
3. **편집**을 클릭합니다.
4. 이름과 설명을 수정합니다.

### 스냅샷 삭제

1. 인스턴스 상세 화면에서 `인스턴스 구성 요소 > 스냅샷`을 선택합니다.
2. 삭제할 스냅샷을 선택합니다.
3. **삭제**를 클릭합니다.

### 스냅샷 롤백

1. 인스턴스가 `종료됨` 상태인지 확인합니다.
2. `인스턴스 구성 요소 > 스냅샷`을 선택합니다.
3. 롤백할 스냅샷을 선택합니다.
4. **롤백**을 클릭합니다.

> 참고
>
> `ISO 파일`로 설치한 가상머신뿐 아니라, Rocky Linux·Ubuntu 같은 Linux 게스트에서도 기본 이미지에 게스트 에이전트가 없을 수 있습니다. 실행 중 스냅샷 정합성이 필요한 경우 QEMU Guest Agent 설치 여부를 확인합니다.

### 게스트 에이전트 설치 가이드

#### Linux 게스트(Rocky Linux / RHEL 계열)

```bash
sudo dnf install -y qemu-guest-agent
sudo systemctl enable --now qemu-guest-agent
sudo systemctl status qemu-guest-agent
```

#### Ubuntu 게스트

```bash
sudo apt-get update
sudo apt-get install -y qemu-guest-agent
sudo systemctl enable --now qemu-guest-agent
sudo systemctl status qemu-guest-agent
```

#### Windows 게스트

Windows 게스트에서는 `virtio-win.iso` 미디어를 사용해 VirtIO 드라이버와 QEMU Guest Agent를 설치합니다.

1. Windows에서 **File Explorer > This PC**로 이동합니다.
2. CD-ROM에 연결된 `virtio-win.iso` 미디어를 엽니다.
3. ISO 루트의 `virtio-win-guest-tools`를 실행합니다.
4. 설치 마법사에 따라 설치를 완료합니다.
5. 설치 후 **Computer Management > Services**에서 `QEMU Guest Agent` 서비스가 `Running` 상태인지 확인합니다.
6. 필요하면 **Device Manager**에서 VirtIO 네트워크, 스토리지 장치 드라이버가 정상 인식되었는지 확인합니다.

#### Windows VirtIO 드라이버 설치 안내

Windows 게스트에서 VirtIO 장치를 정상적으로 사용하려면 `virtio-win.iso`에 포함된 드라이버를 설치해야 합니다. 일반적으로 ISO 루트의 `virtio-win-guest-tools`를 실행하면 주요 VirtIO 드라이버와 QEMU Guest Agent를 함께 설치할 수 있습니다. 특정 장치만 직접 설치해야 하는 경우에는 장치 관리자 또는 Windows 설치/드라이버 선택 화면에서 해당 드라이버 폴더를 지정합니다.

| 구분                     | 파일/경로                        | 설명                            |
| ------------------------ | -------------------------------- | ------------------------------- |
| **통합 설치 파일**       | `virtio-win-guest-tools`         | VirtIO 드라이버 통합 설치       |
| **네트워크 드라이버**    | `NetKVM\<os-version>\amd64`      | VirtIO 네트워크 어댑터 드라이버 |
| **VirtIO-SCSI 드라이버** | `vioscsi\<os-version>\amd64`     | VirtIO-SCSI 디스크 드라이버     |
| **VirtIO-BLK 드라이버**  | `viostor\<os-version>\amd64`     | VirtIO Block 디스크 드라이버    |
| **QEMU Guest Agent**     | `guest-agent\qemu-ga-x86_64.msi` | 게스트 에이전트 개별 설치 파일  |

자주 사용하는 경로 예시는 다음과 같습니다.

| 용도                                         | 경로 예시            |
| -------------------------------------------- | -------------------- |
| **Windows 11 네트워크 드라이버**             | `NetKVM\w11\amd64`   |
| **Windows Server 2022 네트워크 드라이버**    | `NetKVM\2k22\amd64`  |
| **Windows 11 VirtIO-SCSI 드라이버**          | `vioscsi\w11\amd64`  |
| **Windows Server 2022 VirtIO-SCSI 드라이버** | `vioscsi\2k22\amd64` |

> 참고
>
> - 네트워크 드라이버는 `NetKVM`을 사용합니다.
> - 스토리지 드라이버는 디스크 버스 타입에 따라 `vioscsi` 또는 `viostor`를 사용합니다.
> - `virtio-win-guest-tools`를 사용하면 주요 드라이버와 QEMU Guest Agent를 함께 설치할 수 있습니다.
> - 게스트 에이전트만 별도로 설치해야 할 경우에는 `guest-agent` 폴더의 설치 파일을 사용합니다.

#### Windows 11 초기 설정 중 네트워크 연결 문제 해결

Windows 11 설치 중 네트워크 드라이버가 아직 설치되지 않아 진행이 막히는 경우에는 다음 절차로 로컬 계정 기반 설치를 계속할 수 있습니다.

1. 네트워크 연결 화면에서 **Shift + F10**을 눌러 명령 프롬프트(CMD)를 엽니다.
2. 다음 명령어를 입력하고 Enter를 누릅니다.

```cmd
OOBE\BYPASSNRO
```

3. VM이 자동으로 재부팅됩니다.
4. 설치 과정을 다시 진행하면 네트워크 연결 화면에 **인터넷에 연결되어 있지 않음** 옵션이 표시됩니다.
5. 해당 옵션을 선택합니다. 
6. **제한된 설치로 계속**을 선택해 로컬 계정으로 Windows에 진입합니다. 
7. Windows 진입 후 장치 관리자 또는 `virtio-win-guest-tools`를 사용해 `virtio-win.iso`의 VirtIO 드라이버를 설치합니다.

#### Windows 11 디스크 복제 가능 상태 만들기

Windows 11 템플릿 또는 골든 이미지 생성 전에는 BitLocker 자동 암호화 여부를 확인하고, 필요한 경우 복호화가 완료된 뒤 Sysprep을 수행합니다.

1. 관리자 권한으로 명령 프롬프트 또는 PowerShell을 엽니다.
2. BitLocker 상태를 확인합니다.

```cmd
manage-bde -status C:
```

3. 장치 암호화 자동 적용을 방지합니다.

```cmd
reg add "HKLM\SYSTEM\CurrentControlSet\Control\BitLocker" /v PreventDeviceEncryption /t REG_DWORD /d 1 /f
```

4. OS 디스크에 BitLocker가 사용 중이면 복호화를 시작합니다.

```cmd
manage-bde -off C:
```

5. 데이터 디스크가 암호화되어 있으면 같은 방식으로 복호화를 시작합니다.

```cmd
manage-bde -off D:
manage-bde -off E:
```

6. 각 디스크의 복호화가 완료될 때까지 상태를 반복 확인합니다.

```cmd
manage-bde -status C:
manage-bde -status D:
manage-bde -status E:
```

7. BitLocker 보호가 해제되고, 암호화 비율이 0%인지 확인합니다.
8. Sysprep을 실행해 복제 가능한 상태로 정리합니다. 

```cmd
cd \Windows\System32\Sysprep
sysprep.exe /generalize
```

> 참고
>
> - `manage-bde -off`는 BitLocker 보호를 일시 중지하는 것이 아니라 실제 복호화를 시작합니다.
> - 복호화 완료 전에는 스냅샷 생성 또는 이미지 캡처를 진행하지 않습니다.
> - `/generalize`는 SID, 장치 고유 상태, 하드웨어 종속 정보를 정리합니다.
> - Sysprep 이후 인스턴스를 종료하고, 스냅샷 생성 또는 이미지 캡처 전까지 다시 부팅하지 않습니다.

## 추가 블록 디스크와 파일 시스템 연결 확인

추가 블록 디스크나 파일 시스템을 연결한 뒤에는 **화면에서 연결 상태**와 **게스트 OS에서 실제 사용 가능 상태**를 모두 확인합니다.

### 추가 블록 디스크 확인

#### Linux 게스트

1. 인스턴스 상세 화면에서 추가 블록 디스크의 **장치 레이블**을 확인합니다.
2. 게스트 OS에서 다음 명령어를 실행합니다.

```bash
lsblk -o NAME,SERIAL
```

3. 출력 결과에서 각 디스크의 `SERIAL` 값을 확인합니다.
4. **부트 디스크의 serial**과 인스턴스 상세 화면의 **부트 디스크 장치 레이블 숫자(ID)** 가 같은지 확인합니다.
5. 추가 블록 디스크도 같은 방식으로 `SERIAL` 값과 장치 레이블 숫자(ID)를 비교합니다.

| 단계 | 확인 명령 또는 항목                   | 확인 내용                                                    |
| ---- | ------------------------------------- | ------------------------------------------------------------ |
| 1    | 인스턴스 상세 화면의 디스크 정보 확인 | 디스크별 장치 레이블 숫자(ID)를 확인합니다.                  |
| 2    | `lsblk -o NAME,SERIAL`                | 디스크 목록과 `SERIAL` 값이 함께 표시되는지 확인합니다.      |
| 3    | 부트 디스크의 `SERIAL` 값 확인        | 부트 디스크 장치 레이블 숫자(ID)와 일치하는지 확인합니다.    |
| 4    | 추가 블록 디스크의 `SERIAL` 값 확인   | 각 추가 블록 디스크의 장치 레이블 숫자(ID)와 일치하는지 확인합니다. |

> 참고
>
> 디스크가 여러 개 연결된 경우에는 장치 이름(`/dev/vdb`, `/dev/vdc` 등)만으로 식별하지 말고, **`SERIAL` 값과 인스턴스 상세 화면의 장치 레이블 숫자(ID)** 를 함께 비교하십시오.

#### Windows 게스트

1. PowerShell에서 디스크 인식 여부를 확인합니다.

```powershell
Get-Disk
Get-Partition
Get-Volume
```

2. 새 디스크가 `Offline` 또는 `RAW`로 보이면 초기화, 파티션 생성, 포맷 후 드라이브 문자를 할당합니다.
3. 파일 탐색기 또는 `Get-Volume` 출력에서 드라이브 문자가 보이는지 확인합니다.

### 파일 시스템 연결 확인

파일 시스템은 유형에 따라 확인 방법이 다릅니다. 먼저 인스턴스 상세 페이지의 **연결된 리소스 > 파일 시스템**에서 연결 상태를 확인한 뒤, 게스트 OS에서 실제 마운트 상태를 점검합니다.

> 참고
>
> 이 절차는 현재 Linux 게스트(Rocky Linux / RHEL 계열, Ubuntu) 기준으로 정리합니다.

#### Linux 게스트별 준비 사항

| 게스트 OS | NFS 점검 전 확인 사항 | 확인 명령 |
| --------- | --------------------- | --------- |
| **Rocky Linux / RHEL 계열** | `nfs-utils` 패키지가 설치되어 있는지 확인합니다. | `rpm -q nfs-utils` |
| **Ubuntu** | `nfs-common` 패키지가 설치되어 있는지 확인합니다. Ubuntu 기본 이미지에 따라 이 패키지가 없을 수 있습니다. | `dpkg -l nfs-common` |

필요한 패키지가 없으면 다음 명령으로 설치합니다.

```bash
# Rocky Linux / RHEL 계열
sudo dnf install -y nfs-utils

# Ubuntu
sudo apt-get update
sudo apt-get install -y nfs-common
```

#### Linux 게스트 - virtiofs

1. 인스턴스 상세 페이지에서 **연결된 리소스 > 파일 시스템**을 클릭합니다.
2. virtiofs 파일 시스템이 목록에 표시되는지 확인합니다.
3. 게스트 OS에서 다음 명령어를 실행합니다.

```bash
mountpoint /mnt/virtiofs
```

4. 마운트 정보를 확인합니다.

```bash
df -h /mnt/virtiofs
```

5. 테스트 파일을 생성합니다.

```bash
sudo sh -c 'echo "Hello ---" > /mnt/virtiofs/test-file.txt'
```

6. 파일 내용을 확인합니다.

```bash
cat /mnt/virtiofs/test-file.txt
```

7. 파일 목록을 확인합니다.

```bash
ls -la /mnt/virtiofs/
```

| 단계 | 확인 명령 또는 항목                                          | 확인 내용                                   |
| ---- | ------------------------------------------------------------ | ------------------------------------------- |
| 1    | 인스턴스 상세 페이지 → **연결된 리소스 > 파일 시스템**       | virtiofs 파일 시스템이 목록에 표시되는지 확인합니다. |
| 2    | `mountpoint /mnt/virtiofs`                                   | `/mnt/virtiofs`가 마운트 지점으로 인식되는지 확인합니다. |
| 3    | `df -h /mnt/virtiofs`                                        | 파일시스템 용량 정보가 정상적으로 조회되는지 확인합니다. |
| 4    | `sudo sh -c 'echo "Hello ---" > /mnt/virtiofs/test-file.txt'` | 파일 쓰기가 가능한지 확인합니다.             |
| 5    | `cat /mnt/virtiofs/test-file.txt`                            | 파일 읽기가 가능한지 확인합니다.             |
| 6    | `ls -la /mnt/virtiofs/`                                      | 생성한 파일이 목록에 표시되는지 확인합니다. |

> 참고
>
> virtiofs는 마운트 여부뿐 아니라 **읽기/쓰기 테스트까지 수행**하십시오.

#### Linux 게스트 - NFS

1. 게스트 OS에서 `/etc/fstab` 내용을 확인합니다.

```bash
sudo cat /etc/fstab
```

2. 출력 내용에서 NFS 마운트 설정이 있는지 확인합니다.

```text
10.0.10.20:/export/share /mnt/nfs nfs defaults,_netdev,nofail 0 0
```

3. 인스턴스 상세 페이지에서 **연결된 리소스 > 파일 시스템**을 클릭합니다.
4. NFS 파일 시스템이 목록에 표시되는지 확인합니다.

| 단계 | 확인 명령 또는 항목                                    | 확인 내용                                                    |
| ---- | ------------------------------------------------------ | ------------------------------------------------------------ |
| 1    | `sudo cat /etc/fstab`                                  | `fstab` 파일 내용을 확인합니다.                              |
| 2    | fstab 출력 내용에서 NFS 마운트 설정 확인               | NFS 서버 주소, 내보내기 경로, 마운트 경로가 포함된 항목이 등록되어 있는지 확인합니다. |
| 3    | 인스턴스 상세 페이지 → **연결된 리소스 > 파일 시스템** | NFS 파일시스템이 목록에 표시되는지 확인합니다.               |

> 참고
>
> 필요하면 아래 명령어로 실제 마운트 상태를 추가로 점검합니다.
>
> ```bash
> findmnt
> df -hT
> mount | grep nfs
> ```

## 외부 접속 또는 내부 통신이 되지 않을 때 확인할 항목

1. 인스턴스 상태가 `실행 중`인지 확인합니다.
2. 필요한 NIC에 올바른 IP가 설정되어 있는지 확인합니다.
3. 보안그룹 인바운드/아웃바운드 규칙이 허용되어 있는지 확인합니다.
4. 게스트 OS 방화벽과 서비스 리슨 주소를 확인합니다.
5. 플로팅 IP가 필요한 구조라면 올바른 인터페이스에 연결되어 있는지 확인합니다.
6. 다중 NIC 환경이면 `ip route`, `ip rule`, `rp_filter`, 기본 게이트웨이 우선순위와 중복 여부를 확인합니다.

## 인스턴스 가상 CPU 피닝 편집

1. 인스턴스 상세 화면으로 이동합니다.
2. 인스턴스가 `종료됨` 또는 `정지됨` 상태인지 확인합니다.
3. `인스턴스 구성 요소 > 가상 CPU 피닝`을 선택합니다.
4. **편집**을 클릭합니다.
5. 필요한 규칙을 추가하거나 삭제합니다.
6. `vCPU ID`와 `논리 CPU 범위(시작, 끝)`를 입력합니다.
7. **편집**을 클릭해 저장합니다.

> 참고
>
> 가상 CPU 피닝 설정은 다음 실행 시점에 반영되는지 확인한 뒤 작업합니다.
