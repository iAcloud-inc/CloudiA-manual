# 타겟 그룹

`타겟 그룹`은 로드 밸런서가 트래픽을 전달할 대상 집합입니다. 로드 밸런서는 리스너를 통해 타겟 그룹으로 트래픽을 넘기고, 타겟 그룹은 상태 검사 결과와 가중치를 기준으로 실제 대상을 관리합니다.

## 화면 개요
- `목록 화면`: 타겟 그룹 현황을 조회하고 생성, 편집, 삭제 작업을 시작합니다.
- `상세 화면`: 기본 정보, 타겟 목록, 연결된 로드 밸런서 정보를 확인합니다.
- `생성 화면`: 이름, 네트워크, 타겟 유형, 초기 타겟, 상태 검사 설정을 정의합니다.
- `편집 화면`: 이름, 설명, 상태 검사 설정만 수정합니다.
- `타겟 편집 화면`: 상세 화면의 `타겟` 탭에서 대상 목록을 별도로 관리합니다.

## 타겟 그룹 목록
### 주요 작업
- `생성`: 새로운 타겟 그룹을 생성합니다.
- `편집`: 선택한 1개 타겟 그룹의 기본 정보와 상태 검사 설정을 편집합니다.
- `삭제`: 선택한 타겟 그룹을 삭제합니다. 로드 밸런서에 연결되었거나 시스템 생성 그룹이면 삭제할 수 없습니다.

### 테이블 컬럼
| 컬럼           | 설명                                                                                                      |
|--------------|---------------------------------------------------------------------------------------------------------|
| **이름**       | 타겟 그룹 이름입니다. **클릭하면 상세 화면으로 이동**합니다.                                                                    |
| **ID**       | 타겟 그룹 고유 ID입니다.                                                                                         |
| **생성 방식**    | 프로비저닝 소스입니다. 사용자가 직접 생성한 경우 `수동 생성`, 시스템(예: 쿠버네티스, 오토 스케일링 그룹)에서 생성한 경우 `쿠버네티스` 또는 `오토 스케일링 그룹`이 표시됩니다. |
| **VPC**      | 소속 VPC의 이름입니다.                                                                                          |
| **프로토콜**     | `TCP` 또는 `UDP` 입니다.                                                                                     |
| **포트 번호**    | 타겟 그룹 기본 포트입니다.                                                                                         |
| **LB 적용 여부** | 현재 로드 밸런서에 연결되어 있으면 `적용`, 아니면 `미적용`으로 표시됩니다.                                                            |
| **생성 일시**    | 타겟 그룹이 생성된 날짜와 시각입니다.                                                                                               |

## 타겟 그룹 상세
### 상단 요약
- **이름**: 타겟 그룹 이름입니다. 어떤 서비스 대상을 묶은 그룹인지 드러내는 이름을 사용합니다.
- **ID**: 타겟 그룹 고유 ID입니다.
- **생성 방식**: 수동 생성인지, 다른 시스템이 자동 생성한 그룹인지 보여줍니다.
- **VPC**: 타겟 그룹이 속한 VPC입니다.
- **프로토콜**: 타겟 그룹이 처리하는 서비스 프로토콜입니다.
- **포트 번호**: 타겟 그룹의 기본 서비스 포트입니다.
- **타겟 유형**: `인스턴스`, `IP 주소` 중 어떤 방식으로 대상을 등록하는지 보여줍니다.
- **설명**: 운영 메모나 서비스 설명입니다.
- **생성 일시**: 타겟 그룹이 생성된 날짜와 시각입니다.
- **상태 검사 설정**: 헬스 체크 기준을 보여줍니다.
  - **프로토콜**: 상태 검사에 사용하는 통신 프로토콜입니다.
  - **포트 번호**: 상태 검사 포트 번호입니다.
  - **간격**: 상태 검사 간격입니다. (30초)
  - **비정상 임계값**: 연속 실패 허용 횟수입니다. (예: 2 → 두 번 실패 시 트래픽 분산 중단)

### 하단 탭
- **타겟**: 현재 타겟 그룹에 등록된 실제 대상 목록을 확인하는 탭입니다.
- **로드 밸런서**: 이 타겟 그룹을 참조 중인 로드 밸런서 목록을 확인하는 탭입니다.

`타겟` 탭은 실제 트래픽 분산 대상 목록을 표시하는 탭입니다. `로드 밸런서` 탭은 현재 이 타겟 그룹을 참조 중인 로드 밸런서 목록을 표시하는 탭입니다.

### 타겟 탭 컬럼
#### 타겟 유형이 `인스턴스`인 경우
| 컬럼 | 설명 |
| --- | --- |
| **인스턴스** | 타겟으로 등록된 인스턴스 이름과 ID입니다. |
| **IP 주소** | 선택한 인스턴스의 실제 전달 대상 인터페이스 IP 주소입니다. |
| **포트 번호** | 해당 타겟이 실제로 수신하는 서비스 포트 번호입니다. |
| **가중치** | 부하 분산 시 해당 타겟에 적용되는 상대 가중치입니다. |
| **상태** | 상태 검사 결과에 따른 현재 타겟 상태입니다. |

#### 타겟 유형이 `IP 주소`인 경우
| 컬럼 | 설명 |
| --- | --- |
| **IP 주소** | 타겟으로 직접 등록한 VPC 내부 IP 주소입니다. |
| **포트 번호** | 해당 타겟이 실제로 수신하는 서비스 포트 번호입니다. |
| **가중치** | 부하 분산 시 해당 타겟에 적용되는 상대 가중치입니다. |
| **상태** | 상태 검사 결과에 따른 현재 타겟 상태입니다. |

> 💡 **참고:** 상태가 **UNHEALTHY**인 타깃의 경우 트래픽이 분산되지 않습니다.

### 타겟 상태별 의미
| 상태 | 의미 | 분산 대상 포함 여부 | 확인 사항 |
| --- | --- | --- | --- |
| **INITIAL** | 로드 밸런서에 연결된 직후 초기 상태 검사 중인 상태입니다. | 포함되지 않습니다. | 초기 연결 직후 잠시 표시될 수 있으며, 상태 검사 성공 시 `HEALTHY`로 전환됩니다. |
| **HEALTHY** | 상태 검사에 정상 응답한 상태입니다. | 포함됩니다. | 리스너 모니터링과 실제 분산 대상에 반영되는 정상 상태입니다. |
| **UNHEALTHY** | 상태 검사에 실패했거나 응답하지 않은 상태입니다. | 포함되지 않습니다. | 서비스 프로세스, 상태 검사 포트, 보안그룹 인바운드 규칙을 확인해야 합니다. |
| **UNUSED** | 타겟 그룹이 로드 밸런서 리스너에 연결되지 않았거나 연결 해제된 상태입니다. | 포함되지 않습니다. | 타겟 그룹만 생성하고 아직 리스너에 연결하지 않은 경우에도 표시될 수 있습니다. |
| **DRAINING** | 타겟 그룹 연결 해제 과정에서 기존 연결을 정리하는 중간 상태입니다. | 포함되지 않습니다. | 연결 해제 직후 일시적으로 표시될 수 있으며 이후 `UNUSED`로 변경됩니다. |

### 타겟 상태가 `UNHEALTHY`일 때 확인 사항
- 타겟이 `UNHEALTHY`이면 해당 타겟에는 로드 밸런서 트래픽이 분산되지 않으며, 연결된 리스너 모니터링도 기대한 대로 조회되지 않을 수 있습니다.
- 먼저 타겟 그룹의 `상태 검사 프로토콜`과 `상태 검사 포트`가 실제 서비스가 응답하는 프로토콜과 포트인지 확인합니다.
- 가상머신 내부에서 애플리케이션이나 데몬이 실행 중이며, 타겟 그룹에 등록한 포트 또는 상태 검사 포트에서 실제로 응답하고 있는지 확인합니다.
- Cloud:iA 상태 검사는 로드 밸런서의 내부 IP에서 타겟의 상태 검사 포트로 수행됩니다. 보안그룹 인바운드 규칙에서 로드 밸런서 내부 IP 또는 해당 통신을 포함하는 소스 대역이 상태 검사 포트에 대해 허용되어 있는지 확인합니다.
- 서비스 포트와 상태 검사 포트를 다르게 사용한다면, 외부 서비스 포트뿐 아니라 상태 검사 포트도 보안그룹에서 함께 허용되어야 합니다.

### 로드 밸런서 탭 컬럼

| --- | --- |
| **이름** | 이 타겟 그룹을 참조 중인 로드 밸런서 이름입니다. |
| **ID** | 로드 밸런서 고유 ID입니다. |
| **생성 방식** | 로드 밸런서의 프로비저닝 소스입니다. |
| **VPC** | 로드 밸런서가 속한 VPC 이름입니다. |
| **서브넷** | 로드 밸런서가 배치된 서브넷 이름입니다. |
| **외부 IP 주소** | 외부형 로드 밸런서에 할당된 외부 IP 주소입니다. |
| **설명** | 로드 밸런서에 기록된 운영 메모 또는 설명입니다. |
| **생성 일시** | 로드 밸런서가 생성된 날짜와 시각입니다. |

## 타겟 그룹 생성
### 기본 정보
- **이름**: 타겟 그룹을 식별하는 이름입니다.
- **설명**: 어떤 서비스 대상 집합인지 설명합니다.
- **VPC**: 타겟 그룹이 속할 VPC를 선택합니다.
- **프로토콜**: 서비스 통신 프로토콜을 선택합니다.
- **포트 번호**: 타겟 그룹의 기본 서비스 포트를 지정합니다.

### 기본 정보 입력 필드 유효성
| 필드 | 허용 값 | 검증 규칙 | 비고 |
| --- | --- | --- | --- |
| **이름** | 영문, 숫자, `-`, `_` 조합의 문자열 | 필수입니다. 80자 이내로 입력합니다. | 타겟 그룹 식별용 이름입니다. |
| **설명** | 최대 200자 문자열 | 선택 입력입니다. 200자를 초과할 수 없습니다. | 운영 메모용입니다. |
| **VPC** | 현재 프로젝트의 VPC | 필수입니다. 타겟, 로드 밸런서 연결 가능 범위의 기준이 됩니다. | 생성 후 편집할 수 없습니다. |
| **프로토콜** | `TCP`, `UDP` | 필수입니다. 드롭다운에서 선택합니다. | 생성 후 편집할 수 없습니다. |
| **포트 번호** | `1-65535` 정수 | 필수입니다. 기본 서비스 포트이며 유효한 포트 번호여야 합니다. | 상태 검사와 개별 타겟 포트의 기본값으로 사용됩니다. |

`포트 번호`는 상태 검사와 각 타겟에 기본 적용될 디폴트 포트입니다. 필요에 따라 상태 검사 포트와 개별 타겟 포트는 다르게 설정할 수 있습니다.

### 타겟 입력 필드 유효성
| 필드           | 허용 값                              | 검증 규칙 | 비고                                  |
|--------------|-----------------------------------| --- |-------------------------------------|
| **타겟 유형**    | `인스턴스`, `IP 주소`                   | 필수입니다. 타겟 목록에 항목이 하나라도 있으면 생성 화면에서 변경할 수 없습니다. | 생성 후 편집할 수 없습니다.                    |
| **인스턴스**     | 선택한 VPC의 인스턴스                     | `타겟 유형`이 `인스턴스`일 때 필수입니다. | 인스턴스 선택 후 인터페이스 IP를 함께 지정합니다.       |
| **내부 IP 주소** | 선택한 인스턴스의 인터페이스 IP주소 또는 VPC 내부 IP | 필수입니다. 동일한 `IP 주소 + 포트` 조합은 중복 저장할 수 없습니다. | 실제 전달 대상 IP입니다.                     |
| **포트 번호**    | `1-65535` 정수                      | 필수입니다. 유효한 포트 번호여야 합니다. | 기본 포트와 다르게 지정할 수 있습니다.              |
| **가중치**      | `0-100` 정수                        | 필수입니다. 숫자 범위를 벗어날 수 없습니다. | 부하 분산 시 참고 값입니다. 0은 트래픽이 전달되지 않습니다. |

### 타겟 입력 항목
- **타겟 유형**: 대상을 인스턴스 기준으로 등록할지, IP 주소 기준으로 등록할지 결정합니다.
- **인스턴스**: 인스턴스형 타겟일 때 실제 전달 대상 인스턴스를 지정합니다.
- **내부 IP 주소**: 인스턴스형 타겟일 때 인스턴스의 인터페이스 IP주소, IP형 타겟일 때 VPC 내부 IP를 선택합니다.
- **포트 번호**: 실제 서비스가 수신하는 포트를 지정합니다.
- **가중치**: 타겟 간 부하 분산 비중을 조정할 때 사용합니다. 연결하는 로드 밸런서의 부하 분산 알고리즘에 따라 유의미합니다. 

- 인스턴스 타입에서는 인스턴스와 인터페이스 IP를 고릅니다.
- IP 주소 타입에서는 VPC 내부 IP를 직접 선택합니다.
- 타겟은 네트워크 정보가 모두 입력된 뒤에만 추가할 수 있습니다.
- 타겟 목록에 항목이 하나라도 있으면 생성 화면에서 타겟 유형을 바꿀 수 없습니다.

### 타겟 제약
Cloud:iA는 저장 시점의 전체 타겟 목록을 기준으로 중복 여부를 검사합니다. 따라서 새로 추가한 항목끼리뿐 아니라, 기존 항목과 새 항목이 겹치는 경우도 함께 저장할 수 없습니다.

- `인스턴스` 타입은 `선택한 인터페이스 + 포트 번호` 조합이 같으면 중복으로 판단합니다.
- `IP 주소` 타입은 `IP 주소 + 포트 번호` 조합이 같으면 중복으로 판단합니다.
- 같은 인스턴스라도 다른 인터페이스를 선택하면 별도 타겟으로 등록할 수 있습니다.
- 같은 IP 또는 같은 인터페이스라도 포트 번호가 다르면 별도 타겟으로 등록할 수 있습니다.
- 가중치는 0부터 100 사이 숫자를 입력합니다. 0인 경우 트래픽이 전달되지 않습니다. 로드 밸런서의 부하 분산 알고리즘에 따라 유의미합니다.

| 예시 | 결과 | 이유 |
| --- | --- | --- |
| 같은 인스턴스의 같은 인터페이스 `10.0.1.10`에 `8080` 포트를 두 번 등록 | 불가 | `인터페이스 + 포트` 조합이 동일하기 때문입니다. |
| 같은 인스턴스의 같은 인터페이스 `10.0.1.10`에 `8080`, `8443` 포트를 각각 등록 | 가능 | 포트 번호가 다르므로 서로 다른 타겟으로 판단하기 때문입니다. |
| 같은 인스턴스의 서로 다른 인터페이스 `10.0.1.10`, `10.0.2.10`에 같은 `8080` 포트를 등록 | 가능 | 중복 판정 기준이 인스턴스 자체가 아니라 `인터페이스 + 포트` 조합이기 때문입니다. |
| `IP 주소` 타입에서 `10.0.1.20:9000`을 두 번 등록 | 불가 | `IP 주소 + 포트` 조합이 동일하기 때문입니다. |
| `IP 주소` 타입에서 `10.0.1.20:9000`, `10.0.1.21:9000`을 등록 | 가능 | IP 주소가 다르므로 별도 타겟으로 판단하기 때문입니다. |
| `IP 주소` 타입에서 `10.0.1.20:9000`, `10.0.1.20:9001`을 등록 | 가능 | 같은 IP라도 포트 번호가 다르면 별도 타겟으로 판단하기 때문입니다. |

### 가중치가 그룹에서 계산되는 방식
- 타겟 그룹의 `가중치`는 각 타겟이 독립적으로 가지는 절대값이 아니라, 같은 타겟 그룹 안에서 현재 `HEALTHY` 상태인 타겟들끼리 다시 계산되는 상대값입니다.
- Cloud:iA는 전체 타겟을 그대로 계산하지 않고 `HEALTHY` 상태의 타겟만 합산해 그룹 총 가중치를 구합니다.
- 그룹 총 가중치 = `HEALTHY` 타겟들의 가중치 합입니다.
- 각 타겟의 실제 분산 비율은 `해당 타겟 가중치 / 그룹 총 가중치`로 이해하면 됩니다.
- `INITIAL`, `UNHEALTHY`, `UNUSED`, `DRAINING` 상태의 타겟은 그룹 총 가중치 계산과 실제 분산 대상에서 제외됩니다.
- 가중치를 `0`으로 설정하면 `HEALTHY` 상태라도 실제 분산 대상에서 제외하는 용도로 사용할 수 있습니다.

### 계산 예시 1. 모든 타겟이 정상인 경우
다음과 같이 3개의 타겟이 모두 `HEALTHY` 상태라고 가정합니다.

| 타겟 | 상태 | 가중치 |
| --- | --- | --- |
| `web-01` | `HEALTHY` | `50` |
| `web-02` | `HEALTHY` | `30` |
| `web-03` | `HEALTHY` | `20` |

- 그룹 총 가중치는 `50 + 30 + 20 = 100`입니다.
- 실제 분산 비율은 `web-01 50%`, `web-02 30%`, `web-03 20%`로 이해하면 됩니다.
- 즉, `가중 라운드 로빈`, `가중 최소 연결`처럼 가중치를 활용하는 알고리즘에서는 `50:30:20` 비율에 가깝게 분산됩니다.

### 계산 예시 2. 일부 타겟이 비정상인 경우
다음과 같이 `web-02`가 `UNHEALTHY` 상태가 되면 저장된 가중치는 그대로여도 실제 계산 기준이 달라집니다.

| 타겟 | 상태 | 가중치 |
| --- | --- | --- |
| `web-01` | `HEALTHY` | `50` |
| `web-02` | `UNHEALTHY` | `30` |
| `web-03` | `HEALTHY` | `20` |

- `UNHEALTHY` 타겟은 분산 대상에서 제외됩니다.
- 그룹 총 가중치는 `50 + 20 = 70`으로 다시 계산됩니다.
- 실제 분산 비율은 `web-01 약 71.4%`, `web-03 약 28.6%`입니다.
- 즉, 가중치 값 자체를 다시 입력하지 않아도 타겟 상태가 바뀌면 그룹 전체의 실제 분산 비율은 자동으로 다시 계산됩니다.

### 계산 예시 3. 가중치가 0인 타겟이 있는 경우
다음과 같이 설정하면 타겟을 삭제하지 않고도 분산 대상에서 사실상 제외할 수 있습니다.

| 타겟 | 상태 | 가중치 |
| --- | --- | --- |
| `app-01` | `HEALTHY` | `80` |
| `app-02` | `HEALTHY` | `20` |
| `app-03` | `HEALTHY` | `0` |

- 그룹 총 가중치는 `80 + 20 + 0 = 100`입니다.
- 실제 분산 비율은 `app-01 80%`, `app-02 20%`, `app-03 0%`로 이해하면 됩니다.
- 이 방식은 점검 중인 서버를 타겟 그룹에서 삭제하지 않고 일시적으로 분산에서 제외하고 싶을 때 활용할 수 있습니다.

### 가중치와 알고리즘 관계
- `가중 라운드 로빈`, `가중 최소 연결`처럼 가중치를 직접 활용하는 알고리즘에서는 가중치가 큰 타겟이 더 많은 요청을 받을 수 있습니다.
- `라운드 로빈`, `최소 연결`처럼 비가중 알고리즘을 사용할 때는 타겟 간 균등 분산이나 현재 연결 수 기준 분산이 우선입니다.
- `소스 해싱`, `매글레브 해싱`처럼 해시 기반 알고리즘은 연결 일관성이 우선입니다.

### 상태 검사 입력 항목
- **상태 검사 프로토콜**: 타겟 상태를 확인할 때 사용할 프로토콜입니다.
- **상태 검사 포트**: 헬스 체크를 보낼 포트입니다.
- **비정상 임계값**: 연속 실패를 몇 번까지 허용할지 지정합니다.

- 생성 모드에서는 네트워크 정보가 정해지면 상태 검사 프로토콜과 포트의 초기값이 함께 세팅됩니다.
- 상태 검사 간격은 30초로 고정입니다.
- 비정상 임계값은 최소 2 이상입니다.

### 상태 검사 입력 필드 유효성
| 필드 | 허용 값 | 검증 규칙 | 비고 |
| --- | --- | --- | --- |
| **상태 검사 프로토콜** | `TCP`, `UDP` | 필수입니다. 네트워크 정보가 정해지면 초기값이 자동 세팅됩니다. | 타겟 그룹 프로토콜과 함께 검토해야 합니다. |
| **상태 검사 포트** | `1-65535` 정수 | 필수입니다. 유효한 포트 번호여야 합니다. | 초기값은 타겟 그룹 포트입니다. |
| **비정상 임계값** | `2` 이상의 정수 | 필수입니다. 최소 2 이상이어야 합니다. | 연속 실패 횟수 기준입니다. |

## 타겟 그룹 편집
편집 화면에서는 다음 항목을 수정할 수 있습니다.

- **이름**: 타겟 그룹 식별용 이름을 수정합니다.
- **설명**: 운영 메모를 수정합니다.
- **상태 검사 프로토콜**: 헬스 체크 프로토콜을 조정합니다.
- **상태 검사 포트**: 헬스 체크 포트를 조정합니다.
- **비정상 임계값**: 실패 판정 기준을 조정합니다.

다음 항목은 편집할 수 없습니다.

- **VPC**: 생성 후 고정됩니다.
- **프로토콜**: 생성 후 고정됩니다.
- **포트 번호**: 생성 후 고정됩니다.
- **타겟 유형**: 생성 후 고정됩니다.

### 편집 입력 필드 유효성
| 필드 | 허용 값 | 검증 규칙 | 비고 |
| --- | --- | --- | --- |
| **이름** | 영문, 숫자, `-`, `_` 조합의 문자열 | 필수입니다. 80자 이내로 입력합니다. | 편집 가능합니다. |
| **설명** | 최대 200자 문자열 | 선택 입력입니다. 200자를 초과할 수 없습니다. | 편집 가능합니다. |
| **상태 검사 프로토콜** | `TCP`, `UDP` | 필수입니다. 유효한 선택지여야 합니다. | 편집 가능합니다. |
| **상태 검사 포트** | `1-65535` 정수 | 필수입니다. 유효한 포트 번호여야 합니다. | 편집 가능합니다. |

| **포트 번호** | 기존 값 고정 | 편집할 수 없습니다. | 생성 시 결정됩니다. |
| **타겟 유형** | 기존 값 고정 | 편집할 수 없습니다. | 생성 시 결정됩니다. |

### 편집 시 참고 사항
- 타겟 목록 자체는 편집 화면이 아니라 상세 화면의 `타겟` 탭에서 별도로 관리합니다.
- 상태 검사 설정 변경은 타겟의 상태 판정 기준을 바꾸므로 운영 중에는 신중히 반영해야 합니다.

## 타겟 편집
상세 화면 `타겟` 탭에서 진입하는 별도 편집 화면입니다.

### 주요 작업
- `추가`: 타겟을 추가합니다. 
- `편집`: 타겟을 설정을 변경합니다. 
- `삭제`: 타겟을 그룹에서 삭제합니다.

### 편집 화면 특성
- 생성 화면과 동일한 타겟 검증 규칙을 따릅니다.
- 저장 시 현재 화면의 전체 대상 목록이 반영됩니다.

### 타겟 입력 항목
- **인스턴스 / IP 주소**: 인스턴스형 타겟을 수정하거나 추가할 때 대상 인터페이스를 지정합니다.
- **IP 주소**: IP 주소형 타겟을 수정하거나 추가할 때 내부 IP를 지정합니다.
- **포트 번호**: 타겟 서비스 포트를 지정합니다.
- **가중치**: 타겟별 분산 비중을 조정합니다.

### 타겟 입력 필드 유효성
| 필드           | 허용 값                            | 검증 규칙 | 비고 |
|--------------|---------------------------------| --- | --- |
| **인스턴스**     | 선택한 VPC의 인스턴스                   | `타겟 유형`이 `인스턴스`일 때 필수입니다. 동일한 `인스턴스 + 인터페이스 + 포트` 조합은 중복 저장할 수 없습니다. | 생성 화면과 동일합니다. |
| **내부 IP 주소** | 선택한 인스턴스의 인터페이스 IP 또는 VPC 내부 IP | 필수입니다. 동일한 `IP 주소 + 포트` 조합은 중복 저장할 수 없습니다. | 생성 화면과 동일합니다. |
| **포트 번호**    | `1-65535` 정수                    | 필수입니다. 유효한 포트 번호여야 합니다. | 생성 화면과 동일합니다. |
| **가중치**      | `0-100` 정수                      | 필수입니다. 숫자 범위를 벗어날 수 없습니다. | 생성 화면과 동일합니다. |

## 타겟 그룹 삭제
- 여러 개를 선택해 삭제할 수 있습니다.
- 로드 밸런서에 연결된 타겟 그룹은 삭제할 수 없습니다.
- 시스템 생성 타겟 그룹은 삭제할 수 없습니다.

⚠️ **주의:** 타겟 그룹 삭제는 **복구할 수 없습니다.** 삭제 전에 반드시 해당 타겟 그룹의 용도를 확인한 후 삭제하세요.

### 삭제 가능 조건
- `수동 생성` 리소스여야 합니다.
- `로드 밸런서 적용 상태`가 `미적용`이어야 합니다.

### 삭제 전 점검 항목
- 연결된 로드 밸런서가 없는지 확인합니다.
- 오토 스케일링이나 쿠버네티스에서 자동 생성한 타겟 그룹이 아닌지 확인합니다.
